Exemplo Real: Fluxo de Triagem de Candidato — Passo a Passo
Cenário: Um recrutador pede ao agente: "Avalie o candidato João Silva para a vaga de Engenheiro Senior". O que acontece em cada arquitetura? Quais componentes são ativados? Quais proteções são aplicadas?
v5 — Fluxo de Triagem (evaluation domain)
1
HTTP Request chega ao backend
Frontend (Nuxt 3) envia POST /api/chat com a mensagem do recrutador
src/api/routes.py
↓
2
Orchestrator Hub recebe e roteia
orchestrator.py (626L) classifica o intent e identifica: domínio = evaluation
src/orchestrator/orchestrator.py → SupervisorGraph
IntentClassifier DomainCatalog ContextBuilder FairnessGuard PII Masking
↓
3
DomainWorkflow Pipeline (3 nos LangGraph)
classify_intent → execute_action → format_response. Sem fairness check neste pipeline.
src/orchestrator/domain_workflow.py
LangGraph 3 nos GuardrailCheck AuditTrail
↓
4
Evaluation Domain processa
graph.py + nodes.py + state.py — LangGraph interno com InterviewState. Chama Gemini LLM para gerar perguntas e avaliar respostas.
src/domains/evaluation/graph.py → nodes.py → state.py
LangGraph interno InterviewState Gemini LLM FairnessGuard BiasAudit Confidence
↓
5
RubricEvaluation (4 dimensoes)
relevance_score + depth_score + clarity_score + examples_score → overall_score (0-1). Sem framework pedagogico (sem Bloom, sem Dreyfus).
src/domains/evaluation/models.py → RubricEvaluation
4 scores FlowDecision Bloom taxonomy Dreyfus model Big Five
↓
6
Resposta retorna SEM verificacoes
Score + recomendacao ("APPROVED"/"NOT_RECOMMENDED") volta direto ao recrutador. Sem fact-checking, sem bias audit snapshot, sem confidence calibration, sem PII masking dos dados do candidato que foram enviados ao LLM.
src/domains/evaluation/ → response direto
FactChecker BiasAuditSnapshot ConfidenceNode PII demask AuditTrail append-only HITL Gate
RISCOS NESTE FLUXO (v5):
1. Nome, CPF, endereco do candidato foram enviados ao Gemini SEM mascaramento PII
2. Nenhum bias audit — se o LLM discriminou por genero/idade/etnia, nao ha registro
3. Score nao calibrado — sem ConfidenceNode, 0.7 pode significar qualquer coisa
4. Sem auditoria imutavel — registro pode ser alterado/deletado depois
5. Sem fairness check — avaliacao nao passa por FairnessGuard (evaluation e 1 dos 6 dominios sem)
6. Sem fact-checking — claims do candidato nao sao verificados

Identifica dominio = cv_screening. ANTES de processar, verifica guardrails do DB: "Nunca rejeitar sem FairnessGuard" (regra secundaria cv_screening).

EnhancedAgentMixin._setup_enhanced() — 16 capacidades injetadas

libs/agents-core/enhanced_agent_mixin.py

↓
4
FairnessGuard filtra ANTES do LLM
3 camadas: (1) Regex — bloqueia "rejeitar por idade/genero"; (2) Implicito — detecta vieses indiretos ("bairro nobre"); (3) LLM — analise semantica de fairness. Se bloqueado → AuditTrail registra + resposta negada.
app/shared/compliance/fairness_guard.py
Layer 1: regex patterns Layer 2: implicit bias Layer 3: LLM analysis
↓
5
PII Masking remove dados sensiveis ANTES do LLM
4 camadas: CPF → [CPF_MASKED], nome → [NAME_1], endereco → [ADDR_MASKED], telefone → [PHONE_MASKED]. O LLM NUNCA ve dados pessoais reais do candidato.
app/shared/compliance/pii_masking.py
CPF masking Name masking Address masking Phone masking
↓
6
WSI Interview Graph processa (1.141 linhas)
Grafo deterministico com 8 stages: INIT → LOAD_CONTEXT → GENERATE_QUESTION → AWAIT_RESPONSE → VALIDATE → SCORE → ADVANCE → COMPLETE. Cada questao com Bloom (1-6) + Dreyfus (1-5) + Big Five (OCEAN). Estado salvo no PostgresSaver (sobrevive a crashes).
app/domains/cv_screening/agents/wsi_interview_graph.py (1141L)
WSIInterviewState Bloom taxonomy (1-6) Dreyfus model (1-5) Big Five (OCEAN) PostgresSaver checkpoint 3 scores: tech + behav + sit
↓
7
Gemini/Claude processa (com dados mascarados)
LLM recebe texto com [CPF_MASKED], [NAME_1], etc. Anti-sycophancy block no system prompt. CircuitBreaker protege contra falha do provider.
Gemini/Claude API (via CircuitBreaker)
AntiSycophancy block CircuitBreaker ModelRouter
↓
8
FactChecker verifica APOS o LLM
