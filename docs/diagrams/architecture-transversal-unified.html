Recruiter Agent v5 — Arquitetura Completa

1. Como Funciona — Visao Geral
O v5 e um agente conversacional multi-dominio que recebe mensagens de recrutadores via HTTP/WebSocket e as roteia para dominios especializados. A arquitetura segue um modelo de 3 camadas: (1) Hub — orquestracao e sessao, (2) Domains — logica de negocio especializada, (3) Services — infraestrutura transversal. O backend e Python/FastAPI, com Celery para tarefas assincronas, e depende de um backend Rails (ats_api) para persistencia de dados de negocio. Cada dominio e auto-contido: possui seu proprio domain.py, api_client.py, actions/, memory.py e prompts.py.
2. Orquestracao — Hub Architecture (3 Camadas)
Quando uma mensagem chega ao v5, ela passa por um pipeline de roteamento sofisticado:

  ↓ Redis Session Store (ConversationSession por session_id)
HubPlanner (src/hub/planner.py)
  ├── Fast-path: regex patterns → dominio direto (sem LLM, custo zero)
  ├── CostLadder (src/services/cost_ladder.py)
  │   ├── Tier: keywords → domain_id + confidence
  │   ├── Tier: multi-intent detection
  │   └── Tier: LLM fallback (Gemini)
  └── Supervisor LangGraph (src/hub/supervisor_graph.py)
        ↓

  ↓
DomainWorkflow (LangGraph StateGraph: intent → execute → format)
  ↓
Domain (src/domains/{domain}/domain.py: process_intent → execute_action)
  ↓
Rails API (ats_api backend — persistencia de dados)
O HubPlanner e o componente mais inteligente do roteamento: tenta resolver a intencao sem usar LLM (regex → keywords → CostLadder), e so escala para o Gemini quando os tiers anteriores falham. O checkpointing usa PostgresSaver com fallback para MemorySaver — mas o PostgresSaver e instavel (nota 3/10 no audit arquitetural), perdendo estado entre reinicios.
6. Natureza Auto-Contida
O v5 e auto-contido na camada de IA, mas depende fortemente do Rails backend (ats_api) para toda persistencia de dados de negocio. Cada dominio faz chamadas HTTP ao Rails para CRUD de vagas, candidatos, entrevistas etc. Isso cria uma separacao HTTP limpa — o v5 e um "sidecar de IA" do monolito Rails. A vantagem e que o v5 pode ser substituido sem impactar o Rails; a desvantagem e que toda operacao de negocio depende da disponibilidade e latencia do Rails.
7. Pontos Fortes (Pros)
a) Dominios auto-contidos: Cada dominio e um bounded context perfeito — domain.py, api_client.py, actions/, memory.py, prompts.py. Facil de entender, testar e manter isoladamente.
b) Separacao HTTP limpa: v5 ↔ Rails comunicam via HTTP puro, sem dependencia de banco compartilhado. Permite deploy independente.
c) ModelRouter dinamico: Roteamento inteligente fast/default/heavy entre LLMs reduz custos sem sacrificar qualidade.
d) Services consolidados: 51 servicos em uma unica pasta vs 250+ espalhados na LIA — mais facil navegar e encontrar codigo.
e) CostLadder no HubPlanner: Escalonamento progressivo (regex → keywords → LLM) evita chamadas desnecessarias ao LLM.
f) Memoria cross-session madura: TenantMemoryStore com TTL, dedup e busca semantica pgvector e robusto.
g) AppliesDynamicPromptBuilder: Construcao contextual de prompts baseada em actions disponiveis e contexto do job — o componente de prompt mais sofisticado.
8. Pontos Fracos (Contras)
a) Compliance nao obrigatorio: Fairness existe apenas em 2 dos 8 dominios (jobs, sourcing). Os outros 6 dominios (applies, scheduling, evaluation, insights, messaging, autonomous) nao tem nenhuma verificacao de vies. Em producao com clientes brasileiros, americanos ou europeus, isso e risco legal serio.
b) PII vai ao LLM: pii_filter.py mascara CPF/email/phone somente em logs. O texto do usuario vai para o Gemini em texto plano — violacao potencial da LGPD Art. 12 e 46.
c) PostgresSaver instavel (3/10): O checkpointing LangGraph perde estado entre reinicios. Conversas multi-turno (scheduling, evaluation) podem quebrar no meio.
d) Sem regra 4/5 (EEOC): Nao pode fazer auditoria de disparate impact — NYC Local Law 144 (vigente desde 2023) exige isso para sistemas de IA em recrutamento.
e) Sem anti-sycophancy: Os LLMs podem concordar com o recrutador mesmo quando os dados mostram o contrario (ex: "candidato excelente" com score 42/100).
f) Prompts hardcoded em Python: Customizacao por cliente requer alterar codigo e redeploy — impossivel para multi-tenant em escala.
g) Dependencia de 2 times: Qualquer feature que toque negocio requer mudanca no v5 (Python) + Rails (Ruby) — 2 PRs, 2 deploys, coordenacao entre times.
9. Limitacoes de Producao
a) Sem multi-tenant nativo: workspace_id existe em DomainContext mas nao foi projetado para isolamento real. Adicionar multi-tenancy requer modificar todos os 8 dominios + Hub + servicos — estimativa de 2.5 semanas.
b) JSONL audit storage: Auditoria em arquivos JSONL (logs/audit/) e fragil — sem retencao garantida, sem append-only, sem encriptacao. Para SOX/LGPD, precisa migrar para PostgreSQL com retencao de 7 anos.
c) Sem metricas (Prometheus): Nao existe /metrics endpoint — impossivel monitorar latencia, erros e tokens por dominio em producao.
d) Sem DLQ no RabbitMQ: Mensagens que falham sao perdidas — sem dead letter queue para reprocessamento.
10. Limitacoes de Escala
a) 8 dominios fixos: Adicionar um novo dominio requer criar 5+ arquivos (domain.py, api_client.py, actions/, memory.py, prompts.py), registrar no DomainRegistry, e potencialmente criar endpoint no Rails — esforco de 1-2 sprints.
b) Sem tool registry explicito: Tools sao instanciadas dentro de cada dominio sem catalogo central. Quando o agente autonomo precisa de tools de multiplos dominios, a selecao e manual (73+ tools).
c) Memoria siloed: Cada dominio tem sua propria ConversationMemory. Quando o usuario muda de dominio, a memoria anterior nao acompanha — o contexto se perde entre dominios.
d) Escala horizontal limitada: ConversationSession depende de Redis para estado de sessao, mas o PostgresSaver instavel impede checkpointing confiavel em multiplas instancias.
11. Viabilidade para Agent Hub / Agent Studio
Implementar um Agent Hub para clientes (onde empresas configuram seus proprios agentes) no v5 requer:
a) TenantConfigLoader — criar em src/hub/ para carregar configs por workspace_id (nao existe)
b) Override de prompts — prompts sao hardcoded em Python, precisa de proxy decorator por tenant
c) Filtro de tools por tenant — tools instanciadas por dominio, precisa interceptar antes do grafo LangGraph
d) Config vive no Rails — acesso via HTTP interno (latencia adicional)
e) Endpoint Rails + migration — depende de outro time (Ruby)
f) Nunca foi multi-tenant — adicionar workspace_id em todos os pontos e complexidade alta

Esforco estimado: 2.5 semanas (1.5 sprint Python + 1 sprint Rails). Risco de regressao: medio — proxy _TenantOverrideDomain precisa implementar todos os metodos abstratos.

Conclusao Arquitetural
v5 e a escolha quando: dominios auto-contidos sao prioridade, ja existe time Rails dedicado, e compliance pode ser adicionado incrementalmente. Seus pontos fortes sao a clareza estrutural (cada dominio = 1 pasta com tudo dentro) e a eficiencia de roteamento (CostLadder + ModelRouter). Seus riscos sao: PII no LLM, fairness opcional, e dependencia de 2 stacks.

LIA e a escolha quando: compliance regulatorio e inegociavel (LGPD, EU AI Act, NYC LL144), multi-tenant e necessario desde o dia 1, e velocidade de criacao de novos agentes e critica. Seus pontos fortes sao: compliance automatico via heranca (0 linhas por agente), multi-tenant nativo, e Agent Hub viavel em ~2 semanas. Seus riscos sao: overhead de latencia (2-10s), 250+ services desorganizados, e complexidade de debugging da heranca implicita.

Recomendacao para WeDO Talent: usar a LIA como plataforma principal (compliance nao e negociavel em recrutamento), e portar os componentes superiores do v5 (ModelRouter dinamico, AppliesDynamicPromptBuilder, TenantMemoryStore com TTL/dedup) para a LIA. Implementar Agent Hub na LIA primeiro (~2 semanas), validar com 2-3 clientes, depois decidir se o v5 segue como sidecar ou e absorvido pela LIA.
