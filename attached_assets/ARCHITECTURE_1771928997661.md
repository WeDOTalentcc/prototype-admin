# Arquitetura do Sistema - Recruiter Agent V5

**Atualizado em**: 2026-02-10

---

## Visao Geral

Sistema multi-agente para recrutamento e selecao, construido com LangGraph + Google Gemini. Opera em dois modos: **Domain Mode** (queries contextualizadas por dominio) e **Global Mode** (queries genericas sobre a API do ATS). Suporta processamento sincrono (CLI/HTTP) e assincrono (RabbitMQ + Celery).

```
src/
├── agents/              # Agentes do pipeline global (intent, planner, executor, etc)
├── api_controllers/     # Controllers HTTP (evaluation)
├── celery_app.py        # Configuracao Celery com filas por dominio
├── config/              # Settings, configs por servico
├── domains/             # Sistema de dominios (core da arquitetura)
│   ├── base.py          # DomainPrompt (ABC), DomainContext, DomainResponse
│   ├── registry.py      # DomainRegistry + @register_domain decorator
│   ├── orchestrator.py  # DomainOrchestrator
│   ├── workflow.py      # DomainWorkflow (LangGraph: intent → execute → format)
│   ├── base_dispatcher.py  # BaseDispatcher (RabbitMQ consumer base)
│   ├── task_utils.py    # Celery task helpers
│   ├── sourced_profile_sourcing/  # Dominio principal
│   └── evaluation/      # Dominio de avaliacao de candidatos
├── models/              # State models, response models
├── services/            # Servicos compartilhados
├── tools/               # Registry + executor de tools (YAML-based)
├── utils/               # Logger, timing, formatters, callbacks
└── workflow/            # WorkflowOrchestrator global (LangGraph)
```

## Entry Points

### CLI (main.py)

`RecruiterAgentCLI` — modo interativo ou query unica. Usa `WorkflowOrchestrator` direto para queries globais.

`RecruiterAgentWorker` — consumer RabbitMQ que recebe mensagens, monta `context_data` e delega para `MessageRouter`. Suporta dominio via campo `domain` na mensagem ou metadata. Apos processar, envia callback para Rails.

### Celery Workers

Duas filas com prioridades diferentes:

| Dominio | Fila | Prioridade | Tasks principais |
|---------|------|-----------|------------------|
| sourced_profile_sourcing | `sourcing_high` | 10 | `process_query`, `process_query_with_callback`, `refresh_aggregated_stats`, `batch_process_queries` |
| evaluation | `evaluation_normal` | 5 | `process_evaluation`, `evaluate_candidate_with_service`, `consume_and_evaluate`, `process_final_analysis` |

### SourcingDispatcher

Consumer RabbitMQ dedicado (fila `sourcing_requests`) que despacha mensagens para Celery tasks. Recebe JSON com `query`, `user_id`, `callback_url`, `one_time_token`, e envia para `process_query.delay()` ou `process_query_with_callback.delay()`.

---

## MessageRouter

Ponto central de roteamento. Recebe um `dict` com `question`, `domain` (opcional) e `context_data`.

**Com `domain`** → `DomainOrchestrator.process_query(domain_id, question, context_data)`

**Sem `domain`** → `WorkflowOrchestrator.process_query(question)` (pipeline global)

Apos processar, injeta `auth_token` (OTT) e `timing` report na resposta.

---

## Pipeline Global (WorkflowOrchestrator)

Para queries sem dominio. LangGraph com 6 nos:

```
intent_analyzer → api_planner → api_executor → plan_validator → data_processor → answer_formatter
```

- **IntentAnalyzerAgent**: identifica intencao, entities, filtros
- **APIPlannerAgent**: planeja quais APIs chamar (usa YAML tool definitions)
- **APIExecutorAgent**: executa o plano de APIs
- **PlanValidatorAgent**: valida resultados, decide replanejamento (max 3 tentativas)
- **DataProcessorAgent**: agrega e processa dados retornados
- **AnswerFormatterAgent**: formata resposta final em linguagem natural

Edges condicionais: suporta replanning, confirmacao do usuario, abort em falha critica.

## Sistema de Dominios

### DomainPrompt (ABC)

Contrato base que todo dominio implementa:

```python
class DomainPrompt(ABC):
    domain_id: str          # identificador unico
    domain_name: str        # nome legivel
    description: str        # descricao do dominio
    get_allowed_actions()   # lista de DomainAction
    get_system_prompt()     # prompt de sistema
    process_intent()        # analisa intencao do usuario → {action_id, params, confidence}
    execute_action()        # executa acao → DomainResponse
    validate_context()      # valida contexto necessario
    get_suggestions()       # sugestoes para o usuario
```

### DomainRegistry

Registry pattern com decorator `@register_domain`. Armazena classes de dominio e cria instancias sob demanda via `get_instance(domain_id)`.

### DomainContext

Container de contexto compartilhado:

- `domain_id`, `sourcing_id`, `user_id`, `session_id`
- `current_data`, `selected_ids`, `filters_applied`
- `conversation_memory` (lazy-loaded `ConversationMemory`)
- `api_calls_history` (lista de `APICallRecord` para tracking)
- `auth_token`

### DomainResponse

Resposta padronizada com: `success`, `message`, `data`, `suggestions`, `needs_confirmation`, `needs_clarification`, `clarification_question`, `clarification_options`, `error`, `metadata`, `api_calls`.

### DomainWorkflow (LangGraph)

Workflow de 3 nos que processa qualquer dominio:

```
analyze_intent → execute → format
```

**DomainIntentAgent** — Chama `domain.process_intent()` que usa LLM para identificar `action_id`, `params`, `confidence`. Se `needs_clarification=true`, retorna pedido de esclarecimento.

**DomainExecutorAgent** — Valida confidence (>= 0.5). Chama `domain.execute_action()` que delega para o sistema multi-agente (no caso do sourcing) ou processa direto (evaluation).

**DomainResponseBuilder** — Formata resultado via `TemplateFormatter` (Markdown com tabelas, emojis, metricas).

---

## DomainOrchestrator

Coordena o processamento de queries de dominio:

1. Busca dominio no `DomainRegistry`
2. Constroi `DomainContext` a partir de `context_data`
3. Valida contexto (`domain.validate_context()`)
4. Para sourcing: carrega `aggregated_stats` (cache em memoria via `StatsManager`)
5. Delega para `DomainWorkflow.process()`
6. Constroi `DomainResponse` final com timing e metadata
7. Cria mensagem no ATS via API (para sourcing)

## Dominio: sourced_profile_sourcing

### Visao Geral

Dominio principal. Analisa perfis de candidatos vinculados a um sourcing especifico. Requer `sourcing_id` no contexto.

### Actions (30+)

Organizadas por tipo:

| Tipo | Acoes |
|------|-------|
| QUERY | `show_candidate_details`, `search_candidates`, `filter_by_skill`, `filter_by_score`, `list_candidates`, `list_candidates_by_index`, `top_candidates`, `top_by_experience` |
| AGGREGATE | `count_candidates`, `count_by_filter`, `average_score`, `average_experience`, `average_salary_expectation`, `location_distribution`, `gender_distribution`, `language_distribution`, `education_distribution`, `work_model_distribution`, `work_model_specific`, `score_above` |
| ANALYZE | `analyze_skills`, `compare_candidates`, `summarize_search`, `diversity_analysis`, `generate_executive_report`, `generate_top_candidates_report`, `analyze_search_improvement`, `common_strengths`, `skill_gaps`, `candidates_to_discard`, `needs_screening`, `priority_ranking` |

### Fluxo Intent → Execute

`process_intent()` usa LLM (Gemini) com prompt construido pelo `DynamicPromptBuilder`. Recebe a query, lista de actions disponiveis, e aggregated_stats. Retorna JSON com `action_id`, `params`, `confidence`.

`execute_action()` tem handlers mapeados para cada action. Acoes simples (counts, averages, distributions) sao processadas localmente usando `aggregated_stats`. Acoes complexas (search, compare, report) delegam para o `MultiAgentOrchestrator`.

### MultiAgentOrchestrator

Coordena 6 agentes especializados + RouterAgent.

**Fluxo de decisao:**

```
Query chega
  │
  ├─ _analyze_query_complexity() → detecta padrao em QUERY_PATTERNS?
  │   ├─ SIM → cria ExecutionPlan multi-etapa → _execute_plan()
  │   └─ NAO → RouterAgent.process() (query simples)
  │
  └─ Resposta final (DomainResponse)
```

### QUERY_PATTERNS (Planos Multi-Etapa)

8 padroes pre-definidos com regex. Cada padrao gera um pipeline de tasks sequenciais:

| Padrao | Pipeline |
|--------|----------|
| `apply_top_candidates` | search.top → action.create_apply |
| `convert_filtered` | search.filter → action.convert_to_candidate |
| `add_filtered_to_list` | search.filter → action.add_to_list |
| `apply_filtered` | search.filter → action.create_apply |
| `report_filtered` | search.filter → report.generate |
| `compare_filtered` | search.filter → comparison.compare |
| `detail_best` | search.top(limit=1) → detail.show |
| `analyze_skill_holders` | search.filter_skill → analytics.summarize |

`ExecutionPlan` gerencia tasks (`AgentTask`) com status, dependencias, e armazena resultados intermediarios em `context_data` para tasks dependentes.

### RouterAgent

Roteador inteligente com 3 estrategias em cascata:

**1. Memory Patterns** — Detecta referencias conversacionais:
- Pronomes: "e dele?", "fale sobre ela" → resolve `last_candidate_detailed`
- Posicao: "o primeiro", "o terceiro" → resolve por indice em `last_candidates_shown`
- Anterior: "o anterior" → resolve do `detailed_history`
- Shortlist: "salve", "favorite" → add/show shortlist
- Continuacao: "desse grupo" → mantem filtros ativos

**2. Fast Routing (~80% dos casos)** — Keywords + regex patterns com scoring.
- Cada agente tem `keywords` e `patterns` com pesos
- `negative_patterns` para evitar falsos positivos
- Desempate por prioridade: action > detail > comparison > search > report > analytics
- Score minimo de 2 para aceitar

**3. LLM Routing (~5% dos casos)** — Constroi prompt com descricao de todos os agentes e pede ao LLM para decidir. Fallback para `analytics`.

Apos roteamento:
- Extrai params via `ParamExtractor` (regex + `SmartExtractor` com LLM fallback)
- Valida params via `DataValidator` (verifica IDs, resolve nomes com fuzzy matching)
- Delega para agente selecionado
- Atualiza `ConversationMemory` com resultado

### Agentes Especializados

Todos herdam de `BaseAgent`:

```python
class BaseAgent(ABC):
    llm: ChatGoogleGenerativeAI  # lazy-loaded
    validator: DataValidator      # lazy-loaded
    fact_checker: FactChecker     # lazy-loaded

    get_api_client() → SourcingAPIClient
    get_api_operations() → SourcingAPIOperations
    _call_llm(system_prompt, user_message) → str
    _parse_json_response(content) → Dict
    _fetch_candidates(context, limit, order, where) → List[Dict]
    _validate_and_correct_params(params, context) → (success, params, warnings)
    _sanitize(value, max_len) → str

    # abstratos
    agent_id, agent_name, description, capabilities
    get_system_prompt(context)
    process(query, context, aggregated_stats, params) → AgentResponse
```

**AgentResponse** — `success`, `message`, `data`, `suggestions`, `error`, `confidence`, `warnings`. Tem `.to_domain_response()` para converter em `DomainResponse`.

#### SearchAgent (797 linhas)
- Busca por skill, termo livre, score minimo, localizacao
- Top N candidatos por score
- Busca por perfis similares (embeddings)
- Advanced search com multiplos filtros
- `FairnessGuard` para bloquear filtros discriminatorios

#### AnalyticsAgent (592 linhas)
- Contagens (total, por filtro, por score)
- Medias (score, experiencia, salario)
- Distribuicoes (localizacao, genero, skills, idiomas)
- Analise de diversidade
- Sumarizacao geral

#### DetailAgent (301 linhas)
- Perfil completo do candidato
- Analise da IA (Lia)
- Experiencias, skills, alertas
- Resolve identificador por ID, nome, ou memoria

#### ComparisonAgent (873 linhas)
- Compara top N candidatos
- Compara por nomes ou IDs
- Compara candidatos com perfil da vaga
- Gera tabela comparativa + recomendacao via LLM

#### ReportAgent (529 linhas)
- Relatorio executivo completo
- Relatorio focado nos top candidatos
- Usa LLM para gerar insights e recomendacoes
- `FairnessGuard` + anonimizacao para LLM (`anonymize_for_llm`)

#### ActionAgent (728 linhas)
- Converter sourced_profile → candidate
- Criar applies (candidaturas em vagas)
- Adicionar a listas
- Atualizar scores/perfis
- Remover perfis
- Deteccao de intent por regex (`ACTION_PATTERNS`)

### ConversationMemory

Estado conversacional mantido no `DomainContext`:

```
last_candidates_shown: List[int]     # ultima lista exibida (max 20)
last_candidate_detailed: Optional[int]
detailed_history: List[int]          # historico de detalhamentos
shortlist: List[int]                 # favoritos do usuario
mentioned_candidates: Dict[str, int] # nome → id
active_filters: Dict[str, Any]      # filtros ativos
last_search_term: Optional[str]
last_action: Optional[str]
last_job_id: Optional[int]
last_job_name: Optional[str]
```

Metodos de resolucao: `resolve_reference()` tenta pronome → posicao → anterior → nome.

`should_keep_filters()` detecta continuacao de filtro ("desse grupo", "dentre eles").

Serializavel via `to_dict()` / `from_dict()`.

### Servicos de Suporte (sourcing)

| Servico | Descricao |
|---------|-----------|
| `SourcingAPIClient` | HTTP client (httpx) para `/context_for_ai`, `/sourcings/:id`, create_message. Auth via OTT |
| `SourcingAPIOperations` | Camada de alto nivel: search_candidates, get_by_id, get_by_ids, search_by_names, get_top, get_by_score, get_aggregated_stats |
| `DataValidator` | Valida candidate IDs contra sourcing, resolve nomes com fuzzy matching (similarity) |
| `FactChecker` | Verifica claims da IA contra dados reais (contagens, medias, distribuicoes) |
| `StatsManager` | Cache em memoria (thread-safe, TTL, LRU) para aggregated_stats do sourcing |
| `FairnessGuard` | Bloqueia filtros discriminatorios (genero, raca, idade). Compliance LGPD/EEO |
| `ParamExtractor` | Extrai parametros estruturados de queries em linguagem natural via regex |
| `SmartExtractor` | Extracao em 2 estagios: regex rapido + LLM fallback (Gemini). Cache por MD5 |
| `TemplateFormatter` | Formata DomainResponse em Markdown rico (tabelas, emojis, metricas) |
| `DynamicPromptBuilder` | Constroi prompts do sistema e de intencao com documentacao de filtros e acoes |

## Dominio: evaluation

Processa e avalia respostas de candidatos em entrevistas de emprego via chat.

### Fluxo (LangGraph)

```
classify_input → (answer?) → evaluate → decide_flow → craft_message
                  (other?)  → decide_flow → craft_message
```

- **classify_input_node**: Classifica mensagem do candidato (answer/off_topic/unclear). Inclui deteccao de injection.
- **evaluate_response_node**: Avalia qualidade da resposta com scoring via LLM estruturado
- **decide_flow_node**: Decide proximo passo (follow-up, proxima pergunta, encerrar)
- **craft_message_node**: Gera mensagem de resposta/feedback

Usa `MemorySaver` para checkpointing do grafo.

### EvaluationProcessor

Orquestrador que invoca o grafo e converte resultado para `EvaluationResponse` (scores, feedback, chat_ack, follow-up, next_question, etc).

### EvaluationService

Servico alternativo de avaliacao (usado em task `evaluate_candidate_with_service`). Constroi prompts com descricao da vaga, pergunta, resposta, historico. Chama Gemini com retry.

### Tasks Celery

- `process_evaluation` — processa via EvaluationProcessor
- `evaluate_candidate_with_service` — processa via EvaluationService
- `consume_and_evaluate` — processa + publica resultado em fila RabbitMQ de saida
- `process_final_analysis` — analise final do candidato (todas as respostas)
- `publish_evaluation_response` — publica resposta na fila de saida

## Servicos Compartilhados

### ATSAPIClient (src/services/api_client.py)

Client HTTP generico para a API Rails do ATS. Retry com backoff, refresh de token em 401, limpeza de params, paginacao. Carrega endpoints de arquivos YAML via `EndpointLoader`.

### RAGService

Retrieval-Augmented Generation hibrido: `pgvector` para busca semantica + `pg_trgm` para full-text search no PostgreSQL. Usa `EmbeddingService` para gerar embeddings das queries.

### AuthService

Gerencia autenticacao JWT. Login via `/v1/sessions`. Suporta OTT (one-time token) para requests via RabbitMQ. Cache de token com TTL.

### MemoryService

Persistencia em PostgreSQL para historico de conversas (`conversation_history`) e metricas (`query_metrics`).

### RabbitMQService

Publisher/subscriber sobre RabbitMQ com pika. Exchange + queue declarados automaticamente. Mensagens persistentes com ACK/NACK.

## Tools System

### ToolRegistry

Registry singleton thread-safe. Carrega definicoes de tools a partir de arquivos YAML (pasta `documentation/`). Indexa por ID, entity group e category.

### HttpExecutor / WorkflowExecutor

Strategy pattern para execucao de tools. `HttpExecutor` faz chamadas HTTP com substituicao de path params. `WorkflowExecutor` orquestra workflows multi-step.

Carrega `.env` via python-dotenv. Propriedades lazy-loaded para cada modulo:

- `gemini` → `GeminiConfig` (api_key, model: gemini-2.5-flash, temperature)
- `langsmith` → `LangSmithConfig`
- `postgres` → `PostgresConfig`
- `ats_api` → `ATSAPIConfig`
- `rabbitmq` → `RabbitMQConfig`
- `memory` → `MemoryConfig`
- `multi_agent` → `MultiAgentConfig`
- `callback` → `CallbackConfig`
- `rails_api` → `RailsAPIConfig`

### CeleryConfig

Broker: RabbitMQ. Backend: Redis (ou rpc://). Filas com prioridade (x-max-priority=10). Concorrencia configuravel. Soft time limit 240s, hard limit 300s.

### DomainSettings (sourcing)

Configuracoes especificas: limites do orchestrator (max_iterations), memoria (max_history, max_shortlist, max_mentioned), thresholds.

## Fluxo Completo: Query de Sourcing

```
1. Mensagem chega (RabbitMQ/CLI)
   │
2. MessageRouter.route()
   ├── configura OTT (auth)
   ├── extrai question, domain, context_data
   └── domain="sourced_profile_sourcing" → _process_domain_query()
       │
3. DomainOrchestrator.process_query()
   ├── DomainRegistry.get_instance("sourced_profile_sourcing")
   ├── _build_context() → DomainContext
   ├── domain.validate_context() → OK
   ├── domain.ensure_aggregated_stats_sync() → cache/API
   └── DomainWorkflow.process()
       │
4. DomainWorkflow (LangGraph)
   ├── analyze_intent
   │   └── domain.process_intent() → LLM → {action_id, params, confidence}
   ├── execute
   │   └── domain.execute_action() → MultiAgentOrchestrator.process()
   │       ├── _analyze_query_complexity() → padrao ou simples?
   │       ├── (padrao) → ExecutionPlan → tasks sequenciais
   │       └── (simples) → RouterAgent.process()
   │           ├── memory patterns? → resolve referencia
   │           ├── fast route? → keywords/patterns → agente
   │           └── llm route? → LLM decide agente
   │           └── agent.process() → AgentResponse
   └── format
       └── TemplateFormatter.format() → Markdown
           │
5. DomainResponse retorna
   ├── message (formatada)
   ├── data, suggestions, metadata
   ├── timing report
   └── auth_token
```

## Fluxo Completo: Avaliacao de Candidato

```
1. Mensagem do candidato chega (RabbitMQ)
   │
2. Celery task: evaluate_candidate_with_service() ou process_evaluation()
   │
3. EvaluationService.evaluate_candidate_response()
   │   ou EvaluationProcessor.process()
   │
4. LangGraph (evaluation)
   ├── classify_input → answer/off_topic/unclear
   ├── evaluate → scoring + feedback (LLM structured output)
   ├── decide_flow → follow-up? next question? end?
   └── craft_message → resposta ao candidato
   │
5. Resultado publicado na fila RabbitMQ de saida
   └── consume_and_evaluate() → publish para evaluation_responses
```

## Padroes de Design

| Padrao | Onde |
|--------|------|
| Registry | `DomainRegistry`, `ToolRegistry` |
| Strategy | RouterAgent (3 estrategias de roteamento), HttpExecutor/WorkflowExecutor |
| Template Method | `BaseAgent.process()` define esqueleto, subclasses implementam |
| Chain of Responsibility | Memory → Fast → LLM routing |
| Observer | `ConversationMemory` atualizada apos cada agente |
| Decorator | `@register_domain`, `create_domain_task` |
| Singleton | `get_settings()`, `get_validator()`, `get_fact_checker()`, `get_processor()` |
| Builder | `DynamicPromptBuilder` monta prompts |
| Pipeline | LangGraph nodes em sequencia |
| Dispatcher | `SourcingDispatcher`, `BaseDispatcher` (RabbitMQ → Celery) |

## Performance

| Cenario | Estrategia | Tempo Medio |
|---------|-----------|-------------|
| Query simples (fast routing) | keywords/patterns | 300-500ms |
| Query com memoria | resolve referencia | 300-400ms |
| Query complexa (multi-etapa) | ExecutionPlan | 1.5-3s |
| Query ambigua (LLM routing) | Gemini decide agente | 1-2s |
| Intent analysis | Gemini | ~200-400ms |

~80% das queries usam fast routing (sem LLM no roteamento).

Cache de `aggregated_stats` evita recalculos. `SmartExtractor` tem cache por MD5 de query. `ThreadSafeCache` com TTL e LRU eviction.

## Seguranca e Compliance

- **FairnessGuard**: bloqueia filtros por genero, raca, idade, orientacao. Compliance LGPD/EEO
- **Injection Detection**: `safe_process_input()` no dominio evaluation
- **OTT (One-Time Token)**: autenticacao por token descartavel em requests via RabbitMQ
- **Token Refresh**: auto-refresh de JWT em 401
- **Anonimizacao**: `anonymize_for_llm()` remove dados sensiveis antes de enviar ao LLM
