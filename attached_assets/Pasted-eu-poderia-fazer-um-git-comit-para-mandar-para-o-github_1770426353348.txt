🌿 Estratégia 3: Feature Branches (Mais Profissional)
Um Branch por Feature
bash# No Replit

# Criar branch para nova feature
git checkout -b feature/interview-scheduler

# Desenvolver a feature
# ... código, testes, etc ...

# Commitar tudo da feature
git add features/interview-scheduler/
git commit -m "feat(interviews): Smart interview scheduler"

# Push do branch
git push origin feature/interview-scheduler

# Kilo detecta novo branch e cria cards específicos dessa feature
Configurar Kilo para Branches
yaml# .kilo/workflows/feature-branch-to-jira.yaml
name: Feature Branch to Jira Cards
trigger: 
  - on_branch_push
  - pattern: "feature/*"

steps:
  1_detect_feature:
    action: extract_feature_name_from_branch
    # feature/interview-scheduler → "interview-scheduler"
    
  2_analyze_feature:
    tool: get_replit_components
    params:
      path: "features/{feature_name}/"
      
  3_read_config:
    action: read_file
    path: "features/{feature_name}/feature.config.json"
    
  4_create_jira_epic:
    tool: create_jira_epic_from_prototype
    params:
      epicName: "{config.name}"
      components: "{step_2_output}"
      domain: "{config.domain}"
      labels: "{config.jira.labels}"
      
  5_notify:
    action: slack_notify
    message: "Created Jira Epic for feature: {config.name}"
    link: "{epic_url}"
Vantagens:

✅ Isolamento total por feature
✅ Fácil de revisar (1 branch = 1 feature)
✅ Pode testar separadamente
✅ Merge quando pronto

Múltiplas Features Simultaneamente

# Trabalhar em cada um independentemente

# ... desenvolver ...

git push

cd ../candidate-filter
# ... desenvolver ...

Quando usar: Se você trabalha em múltiplas features ao mesmo tempo.

🔄 Kilo Detectando Features Específicas
Configuração para Detectar Apenas Mudanças de Features
yaml# .kilo/config.yaml
monitoring:
  watch_paths:
    - "features/*/"        # Só monitora features/
    - "!features/*/wip/*"  # Ignora subpastas WIP
    
  ignore_paths:
    - "design-system/"     # Ignora design system
    - "shared/"            # Ignora shared (não gera cards)
    - "*.test.js"          # Ignora testes
    - "*.stories.js"       # Ignora storybook

feature_detection:
  require_config: true      # Só processa se tiver feature.config.json
  status_filter: 
    - "ready-for-jira"      # Só cria cards se status = ready-for-jira
    
jira_creation:
  mode: "per-feature"       # Um epic por feature
  auto_create: true         # Cria automaticamente
  require_approval: true    # Mas pede aprovação antes
Skill Customizado: Feature-Specific Jira
yaml# .kilo/skills/feature-to-jira.md
name: Feature-Specific Jira Card Creator
trigger: 
  - git_push
  - contains: "features/"

workflow:
  1. Detect changed feature:
     - Parse git diff
     - Identify which feature/ folder changed
     - Read feature.config.json
     
  2. Validate feature is ready:
     - Check status == "ready-for-jira"
     - Check all required fields exist
     - Verify components are complete
     
  3. Create Jira cards:
     - IF feature.jira.createEpic:
         Create epic with feature name
     - FOR each component:
         Create story linked to epic
     - Use metadata from feature.config.json
     
  4. Link commit to cards:
     - Add commit SHA to Jira cards
     - Add Jira links to commit message (via git notes)
     
  5. Notify:
     - Slack message with Jira links
     - Update feature.config.json with Jira keys

example:
  input:
    git_commit: "feat(interviews): Smart scheduler"
    changed_files: 
      - "features/interview-scheduler/InterviewScheduler.jsx"
      - "features/interview-scheduler/feature.config.json"
      
  output:
    jira_cards:
      - key: "WEDO-1240"
        type: "Epic"
        summary: "Smart Interview Scheduler"
      - key: "WEDO-1241"
        type: "Story"
        summary: "Implement InterviewScheduler component"

🚀 Script de Automação Completo
bash# ~/scripts/new-feature.sh
#!/bin/bash

# Script para criar nova feature no Replit
# Uso: ./new-feature.sh interview-feedback "Automated Interview Feedback" interviews

FEATURE_SLUG=$1
FEATURE_NAME=$2
DOMAIN=$3

if [ -z "$FEATURE_SLUG" ] || [ -z "$FEATURE_NAME" ] || [ -z "$DOMAIN" ]; then
  echo "Uso: ./new-feature.sh <slug> <name> <domain>"
  echo "Exemplo: ./new-feature.sh interview-feedback 'Automated Interview Feedback' interviews"
  exit 1
fi

FEATURE_PATH="features/$FEATURE_SLUG"

# Criar branch
git checkout -b "feature/$FEATURE_SLUG"

# Criar estrutura
mkdir -p "$FEATURE_PATH"/{components,hooks,tests}

# Criar README
cat > "$FEATURE_PATH/README.md" << EOF
# $FEATURE_NAME

## Descrição
[Descrever a feature aqui]

## Componentes
- TBD

## Integração
- APIs: TBD
- Agents: TBD

## Status
- [ ] Protótipo funcional
- [ ] Testes completos
- [ ] Documentação
- [ ] Pronto para Jira
EOF

# Criar feature.config.json
cat > "$FEATURE_PATH/feature.config.json" << EOF
{
  "name": "$FEATURE_NAME",
  "version": "0.1.0",
  "status": "in-progress",
  "domain": "$DOMAIN",
  "type": "feature",
  
  "jira": {
    "createEpic": true,
    "priority": "medium",
    "labels": []
  },
  
  "components": [],
  
  "integration": {
    "apis": [],
    "agents": [],
    "services": []
  },
  
  "changelog": [
    "$(date +%Y-%m-%d): Initial setup"
  ]
}
EOF

# Criar componente inicial
cat > "$FEATURE_PATH/components/${FEATURE_NAME// /}.jsx" << EOF
import React from 'react';

export default function ${FEATURE_NAME// /}() {
  return (
    <div className="p-4">
      <h1>$FEATURE_NAME</h1>
      <p>TODO: Implement feature</p>
    </div>
  );
}
EOF

# Commit inicial
git add "$FEATURE_PATH"
git commit -m "feat($DOMAIN): Initialize $FEATURE_NAME feature"
git push origin "feature/$FEATURE_SLUG"

echo "✓ Feature branch created: feature/$FEATURE_SLUG"
echo "✓ Files created in: $FEATURE_PATH"
echo ""
echo "Next steps:"
echo "1. Develop components in $FEATURE_PATH/components/"
echo "2. Update feature.config.json when ready"
echo "3. Set status to 'ready-for-jira'"
echo "4. Push to trigger Jira card creation"

✅ Resumo da Resposta
Sim, você pode commitar só uma funcionalidade:

Método simples: git add features/nome-feature/ (staging seletivo)
Método recomendado: Estrutura modular + feature branches
Kilo detecta: Apenas a feature commitada e cria cards específicos dela

Benefícios:

✅ Cards Jira focados (1 feature = 1 epic)
✅ Histórico limpo (1 commit = 1 feature)
✅ Time pega features isoladas
✅ Não mistura funcionalidades diferentes
✅ Fácil de revisar e testar
