  2. GAPS CRÍTICOS

  GAP A — Sem modelo de Placement (CRÍTICO)
  Não há tabela para persistir salário final, data de início, bônus, tipo de contrato. A "contratação" hoje é apenas a transição de stage do candidato para "hired". Impede: analytics
  de compensation, reporting, ATS sync completo, integração HRIS, auditoria SOX.

  GAP B — Notificação a candidatos na pausa não está wired
  JobStatusModal tem o checkbox e o template selector, mas bulk/pause ignora esses parâmetros. Candidatos em processo não recebem aviso quando a vaga é pausada.

  GAP C — Sem motivo persistido na pausa e no cancelamento
  Audit log registra que o status mudou, mas não "por que". Campos pause_reason e cancel_reason não existem no modelo. Afeta rastreabilidade e compliance para clientes regulados (BCB
  498/SOX).

  GAP D — Sem fluxo de cancelamento dedicado no FE
  Não há modal de cancelamento. A opção "Cancelada" aparece só em mock data (jobs2-page.tsx). Recrutador não tem forma guiada de cancelar uma vaga com motivo.

  GAP E — Dados de placement não capturados no fechamento
  CandidateDecisionFlowModal.confirm_hire é placeholder (menciona "onboarding" mas sem lógica). CloseVacancyModal envia notificação mas não coleta salário, data, bônus. O endpoint
  POST /close não persiste esses dados.

  GAP F — Feedback construtivo não é personalizado por IA
  CloseVacancyModal usa um template único para todos os candidatos. CandidateFeedbackService tem send_gate_feedback() funcional mas não gera mensagem individual por candidato no
  fechamento de vaga.

  GAP G — ATS sync no hire não é automático via Kanban
  POST /handle-trigger/candidate-hired faz ATS sync, mas é um trigger separado. O fluxo natural (arrastar para Contratado → PATCH /candidates/{id}/stage) não dispara ATS sync
  automaticamente.

  O PROBLEMA DA LISTA DE CANDIDATOS NO MODAL

  O que existe hoje no CloseVacancyModal

  O modal atual mostra uma lista com checkbox por candidato. Isso funciona para vagas com 5-15 pessoas, mas quebra completamente com 50, 200 ou 500 candidatos.

  Ninguém vai revisar 500 nomes num modal. A experiência seria inutilizável.

  ---
  Como o mercado resolve

  Greenhouse, Lever, Gupy não mostram lista individual no momento do fechamento. O que mostram é:

  - Um contador: "487 candidatos receberão esta mensagem"
  - Um resumo por stage: "30 em Triagem · 15 em Entrevista RH · 5 em Short List · 200 em Funil"
  - A ação se aplica a todos os que não estão em stage terminal (reprovado/desistiu/contratado)

  A seleção individual só aparece em casos muito específicos, como "rejeitar candidatos de uma etapa específica" — não no fechamento geral.

  ---
  As três ações possíveis na lista

  1. "Excluir uma etapa" — dropdown para excluir candidatos de um stage inteiro
  ex: "Não enviar para candidatos do Funil/Sourcing" — faz sentido quando são leads frios que nunca interagiram de verdade

  2. "Ver lista →" — abre um painel lateral (drawer) com a lista paginada, apenas para conferência. Não é necessário para enviar — é opcional.

  3. Nada — confirma para todos. Que é o caminho feliz da maioria dos casos.

  Modo Automático vs Manual — como fica

  ┌──────────────────────┬───────────────────────────────────────────────────────────────────┬─────────────────────────────────────────────────────────────┐

  │ O que acontece       │ LIA gera mensagem individual por candidato em background (Celery) │ Mesma mensagem para todos, com {nome} e {vaga} substituídos │
  ├──────────────────────┼───────────────────────────────────────────────────────────────────┼─────────────────────────────────────────────────────────────┤
  │ Exibe lista?         │ Não — só o contador e resumo por stage                            │ Não — só o contador e resumo por stage                      │
  ├──────────────────────┼───────────────────────────────────────────────────────────────────┼─────────────────────────────────────────────────────────────┤
  │ Quando faz sentido   │ Processo seletivo de qualidade, poucos candidatos chegaram longe  │ Alto volume, funil amplo, candidatos de stages iniciais     │
  └──────────────────────┴───────────────────────────────────────────────────────────────────┴─────────────────────────────────────────────────────────────┘

  No modo automático, o recrutar não precisa ver nada — ele confia que a LIA vai personalizar. O trabalho pesado vai para o background.

  ---
  Resumo da decisão de UX

  - Nunca mostrar lista individual no modal de fechamento — impraticável acima de 20 candidatos
  - Mostrar: contador + breakdown por stage — informativo, rápido, escala para qualquer volume
  - Dar controle por stage, não por indivíduo — "excluir sourcing" é suficiente para 99% dos casos
  - Ver lista individual é função de relatório/dashboard, não do modal de fechamento

  O QUE FICA CLARO COM ESTE MODELO

  O painel de placement/fechamento é um wizard conversacional com 3 steps sequenciais. Não pergunta sobre canal — delega isso ao modal de envio existente.

  Pausa e cancelamento seguem o mesmo padrão de painel lateral simples (1 step): motivo + opção de notificar → se notificar, abre modal de envio existente.
