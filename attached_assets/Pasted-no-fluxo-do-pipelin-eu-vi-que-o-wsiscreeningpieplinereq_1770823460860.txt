Como a senioridade chega ao WSI Pipeline
Existem 3 caminhos pelos quais a senioridade é definida, e todos têm o mesmo problema:
Caminho 1: Job Wizard (fluxo conversacional)
1. O recrutador digita algo no chat como "Preciso de um dev senior"
2. O LLM extrai entidades e identifica seniority: "senior"
3. O sistema normaliza via normalize_seniority() que faz um mapeamento simples de strings: "sr" → "Sênior", "junior" → "Júnior", etc.
4. Esse valor é salvo no JobVacancyState.seniority
5. Quando as perguntas WSI são geradas, há outro mapeamento que converte de volta: "Sênior" → "senior", "Júnior" → "junior"
6. Esse "senior" final vai para SENIORITY_TO_BLOOM e SENIORITY_TO_DREYFUS
Caminho 2: Endpoint direto (API)
1. O frontend envia seniority: "pleno" diretamente no request do WSIScreeningPipelineRequest
2. Não há validação — é o que o recrutador selecionou em um dropdown
Caminho 3: Endpoint do Wizard
1. seniority_level: "pleno" vem no GenerateJobScreeningQuestionsRequest
2. Também é input direto do recrutador
O problema que você identificou
Você tem razão: a senioridade é um input "cru" do recrutador. Não existe:
* Nenhuma inferência baseada em anos de experiência
* Nenhuma validação contextual (área, mercado, país)
* Nenhuma calibração — se o recrutador diz "senior", o sistema aceita e gera perguntas Bloom nível 4-5 e Dreyfus nível 4 (Proficiente), independentemente de ser senior em tech (3-5 anos) ou senior em advocacia (8-10 anos)
* O default é "pleno" quando não é informado — ou seja, se o recrutador não mencionar, todas as perguntas ficam no nível médio
Tem ainda outro detalhe importante: os mapeamentos estão duplicados e inconsistentes pelo código. Existem pelo menos 3 dicionários diferentes de normalização em arquivos diferentes, com variações nos valores aceitos.
Conclusão
A senioridade hoje é essencialmente um campo de texto livre que o recrutador preenche (via chat ou dropdown), sem nenhuma inteligência por trás. O sistema simplesmente mapeia esse texto para níveis fixos de Bloom e Dreyfus. Não há consideração de contexto (área, país, mercado) e o default silencioso para "pleno" pode estar distorcendo perguntas sem que o recrutador perceba.
Isso se conecta diretamente à sua pergunta anterior sobre calibração de senioridade — os dois problemas são o mesmo: a falta de contextualização do nível de complexidade das perguntas.
Quer que eu proponha um plano para resolver isso junto com a calibração de senioridade? Poderia ser um sistema que considere o contexto da vaga (área, título, localização) para validar e ajustar a senioridade informada pelo recrutador, em vez de simplesmente aceitá-la como verdade absoluta.
