Proposta de Arquitetura - WedoTalent Admin
Baseado na análise do sistema existente, segue uma proposta completa para a área administrativa:

1. Estrutura de Navegação (Sidebar)
📊 OVERVIEW
├── Dashboard Geral
└── Métricas da Plataforma
👥 GESTÃO DE CLIENTES
├── Lista de Clientes (com busca/filtros)
└── [Seletor de Cliente Ativo] ← Dropdown global no topo
📋 OPERAÇÕES DO CLIENTE (contexto do cliente selecionado)
├── Visão Geral
├── Usuários & Permissões
├── Setup Empresa ✓ (já existe)
├── Jornada de Recrutamento ✓ (já existe)
├── Comunicações ✓ (já construído - 7 abas)
├── Integrações ✓ (já existe)
├── Automações
├── Big Five & Arquétipos
└── Testes Técnicos
💰 FATURAMENTO & MÉTRICAS
├── Planos & Assinaturas
├── Faturamento
├── Consumo de IA (créditos LIA)
└── Métricas SaaS por Cliente
🛠️ OPERAÇÕES & SUPORTE
├── Implementações (onboarding)
├── Incidentes & Problemas
├── Chamados/Tickets (HubSpot?)
├── Monitoramento
└── Observabilidade
⚙️ CONFIGURAÇÕES GLOBAIS (plataforma)
├── Templates Padrão
├── Políticas Globais
├── Configurações do Sistema
└── Auditoria & Logs

2. Arquitetura de URLs
Área	URL	Descrição
Lista de Clientes	/admin/clientes	Listagem com busca/filtros
Perfil do Cliente	/admin/clientes/[clientId]	Overview do cliente
Usuários	/admin/clientes/[clientId]/usuarios	Gestão de usuários do cliente
Comunicações	/admin/clientes/[clientId]/comunicacoes	Aproveita o módulo existente
Jornada	/admin/clientes/[clientId]/jornada	Aproveita o módulo existente
Setup	/admin/clientes/[clientId]/setup	Aproveita o módulo existente
Faturamento	/admin/clientes/[clientId]/faturamento	Billing do cliente
Métricas	/admin/clientes/[clientId]/metricas	Métricas SaaS do cliente
Implementação	/admin/clientes/[clientId]/implementacao	Progresso de onboarding
Suporte	/admin/clientes/[clientId]/suporte	Tickets e incidentes
3. Novos Modelos de Dados Necessários
Modelo	Campos Principais	Propósito
ClientAccount	id, name, cnpj, status, plan_id, created_at	Cliente/empresa contratante
SubscriptionPlan	id, name, price, features, limits	Planos de assinatura
BillingInvoice	id, client_id, amount, status, due_date	Faturas
AiConsumption	id, client_id, type, tokens, cost, date	Consumo de IA
OnboardingChecklist	id, client_id, step, status, completed_at	Progresso de implementação
SupportTicket	id, client_id, subject, status, priority	Chamados (ou integração HubSpot)
Incident	id, client_id, severity, description, status	Incidentes técnicos
ClientUser	id, client_id, user_id, role, permissions	Usuários do cliente

5. Plano de Implementação (Fases)
Fase 1: Fundação (2-3 semanas)
 Criar modelo ClientAccount no backend
 Implementar seletor global de cliente no header
 Criar página /admin/clientes com listagem
 Criar layout /admin/clientes/[clientId] com sidebar contextual
 Implementar context provider para cliente selecionado
Fase 2: Operações Core do Cliente (2-3 semanas)
 Montar módulos existentes dentro do contexto do cliente
 Implementar gestão de usuários (ClientUser + CRUD)
 Criar dashboard de visão geral do cliente
 Adaptar integrações para contexto multi-tenant
Fase 3: Faturamento & Métricas (2-3 semanas)
 Implementar modelos de billing (plans, invoices)
 Criar dashboard de consumo de IA
 Implementar métricas SaaS por cliente
 Criar relatórios de faturamento
Fase 4: Operações & Suporte (2-3 semanas)
 Implementar tracker de onboarding/implementação
 Criar sistema de incidentes
 Integrar com HubSpot para tickets (ou criar interno)
 Dashboards de observabilidade
Fase 5: Configurações Globais (1-2 semanas)
 Templates padrão da plataforma
 Políticas globais (já parcialmente construído)
 Configurações avançadas de governança
 Sistema de auditoria
6. Isolamento Multi-Tenant
┌─────────────────────────────────────────┐
│     Header: [Seletor de Cliente ▼]      │
├─────────────────────────────────────────┤
│                                         │
│  Sidebar        │    Conteúdo           │
│  (contextual    │    (filtrado por      │
│   ao cliente)   │     company_id)       │
│                 │                       │
└─────────────────────────────────────────┘

Todas as APIs recebem X-Company-ID via headers
Backend valida acesso (RBAC com roles: platform_admin, client_admin, user)
Frontend armazena cliente selecionado em Context, refetch ao trocar
