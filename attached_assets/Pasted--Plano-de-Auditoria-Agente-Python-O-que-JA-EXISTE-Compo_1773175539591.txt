## GAPS — O que FALTA
 
### GAP 1 — ReAct agents não propagam AuditCallback [CRÍTICO]
 
**Problema:** O `AuditCallbackHandler` é injetado no `DomainOrchestrator.process_query()` via `config["callbacks"]`, mas os ReAct agents (`AppliesReActAgent`, `UniversalReActAgent`) fazem `graph.invoke(initial_state)` **sem passar config/callbacks**. Resultado: nenhuma chamada LLM ou tool dentro dos agentes autônomos é auditada.
 
**Onde:**
- `src/domains/applies/react_agent.py` L376: `graph.invoke(initial_state)` — sem config
- `src/domains/autonomous/agent.py` L170: `graph.invoke(initial_state)` — sem config
 
**Como o audit chega no workflow normal:**
- `src/domains/orchestrator.py` L95-116: cria `AuditCallbackHandler`, injeta em `config["callbacks"]`, passa para `self.workflow.process()`
- `src/domains/workflow.py` L461: `self.graph.invoke(initial_state, config=config)` — config vem do orquestrador com callbacks
 
**Fix:** Receber callbacks do orquestrador e passar para `graph.invoke(initial_state, config={"callbacks": callbacks})`.
 
**Esforço:** Baixo — 2-3h
 
---
 
### GAP 2 — Autonomous agent não rastreia tools chamadas [MÉDIO]
 
**Problema:** O `AuditCallbackHandler.on_tool_start()` captura o nome da tool, mas o nó `call_tools` do autonomous agent não registra quais tools específicas foram chamadas no tracking dict. Resultado: o log final mostra count total mas não quais tools (ex: "list_departments → get_department_tree → create_department").
 
**Onde:** `src/domains/autonomous/agent.py` L102-120: `call_tools` incrementa counter mas não guarda nomes
**Mesmo problema em:** `src/domains/applies/react_agent.py` L342-353
 
**Fix:** Adicionar `tracking["tools_used"]` como lista que acumula `tc["name"]` a cada tool call. Incluir no metadata da DomainResponse.
 
**Esforço:** Baixo — 1h
 
---
 
### GAP 3 — Sem log pesado em storage externo (S3/GCS) [BAIXO]
 
**Problema:** O `AuditWriter` salva no PostgreSQL com timeline JSONB, mas trunca dados (user_query limitado a 500 chars, tool outputs não são salvos). Não existe camada de storage para prompts completos, respostas brutas e state snapshots para investigação profunda.
 
**Onde:** `src/services/audit/audit_writer.py` — só salva metadata + timeline compacta
**Proposta do doc de arquitetura:** S3/GCS para logs pesados, PG para metadata leve
 
**Fix:** Criar `AuditStorageWriter` que grava JSONL completo em GCS bucket (ou filesystem local em dev). Conectar ao `AuditCallbackHandler.finalize()`. Incluir: prompt completo, todas tool responses, state final.
 
**Esforço:** Médio — 4-6h (inclui setup de bucket, credenciais, retry)
 
**Prioridade:** Baixa agora, útil quando tiver volume em produção. Pode ser adiado.
 
---
 
### GAP 4 — Sem endpoint de reconstrução de timeline [BAIXO]
 
**Problema:** Os dados de audit estão no PG (`agent_executions` table) mas não existe endpoint REST para consultar/filtrar execuções e reconstruir a timeline de uma sessão.
 
**Onde:** Não existe
 
**Fix:** Criar endpoints no FastAPI/main:
- `GET /v1/agent/executions?session_id=X` — lista execuções de uma sessão
- `GET /v1/agent/executions/{execution_id}` — detalhes com timeline completa
- `GET /v1/agent/executions/{execution_id}/timeline` — timeline formatada
 
**Esforço:** Médio — 3-4h
 
**Prioridade:** Baixa — útil para debugging, não é blocker.
 
---
 
### GAP 5 — Sem métricas Prometheus/estruturadas [BAIXO]
 
**Problema:** Não existe exportação de métricas do agente em formato Prometheus (requests/s, latência por domínio, tool calls/s, erros/s, token usage). O tracking existe via logs mas não em formato scrapeable.
 
**Onde:** Não existe
 
**Fix:** Adicionar `prometheus_client` com counters/histograms:
- `agent_requests_total{domain, status}`
- `agent_duration_seconds{domain}`
- `agent_tool_calls_total{tool_name, domain}`
- `agent_llm_tokens_total{type=input|output, model}`
- `agent_errors_total{domain, error_type}`
Expor em `/metrics` endpoint.
 
**Esforço:** Médio — 4-5h
 
**Prioridade:** Baixa — útil quando escalar para produção com Grafana.
 
---
 
### GAP 6 — Retention/cleanup de agent_executions [BAIXO]
 
**Problema:** A tabela `agent_executions` cresce indefinidamente. Não existe job de limpeza.
 
**Onde:** `src/services/audit/audit_writer.py` — cria a tabela mas nunca limpa
 
**Fix:** Criar script/job que roda periodicamente (cron ou Celery beat):
- Deletar execuções com mais de 90 dias
- Mover para storage frio (S3) antes de deletar, se GAP 3 estiver implementado
 
**Esforço:** Baixo — 2h
 
---
 
### GAP 7 — Autonomous agent sem circuit breaker [MÉDIO]
 
**Problema:** O `CircuitBreaker` é usado no `DomainWorkflow._execute_with_retry()` para proteger chamadas de API dos domínios tradicionais. O autonomous agent (`UniversalAPIClient`) faz chamadas HTTP diretas sem circuit breaker — se a API Rails estiver down, o agente vai gastar todas as iterações batendo em erros 5xx.
 
**Onde:**
- `src/domains/workflow.py` L178-220: CircuitBreaker para domínios tradicionais
- `src/domains/autonomous/api_client.py` L50-100: nenhuma proteção
 
**Fix:** Integrar `CircuitBreaker` no `UniversalAPIClient.request()`. Se o circuito estiver aberto, retornar `APIResponse(success=False, error="circuit_open")` imediatamente sem fazer HTTP call.
 
**Esforço:** Baixo — 2h
