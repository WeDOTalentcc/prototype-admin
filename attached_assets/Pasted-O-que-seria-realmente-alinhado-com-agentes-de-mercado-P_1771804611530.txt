O que seria realmente alinhado com agentes de mercado
Para a LIA funcionar como um agente inteligente de verdade no wizard, precisaria de:

ReAct Loop (Reasoning + Acting) — Ao invés de pipeline linear, um loop: Pensar → Decidir Ação → Executar → Observar → Pensar novamente. O LangGraph suporta nativamente.

Working Memory (Scratchpad) — Um estado rico que o agente mantém: "Estou na criação de vaga X, já coletei título/competências/salário, o recrutador pediu para voltar ao salário, preciso mostrar o que já tem e perguntar o que quer mudar"

LLM-based Intent + Routing — Eliminar regex. Cada mensagem vai ao Claude com contexto completo e ele decide: é um ajuste? uma pergunta? quer avançar? quer voltar? quer falar de outra coisa?

Stage-aware Prompt Injection — Ao invés de templates estáticos, o prompt do agente recebe instruções dinâmicas por etapa: "Você está na etapa Salário. O salário foi definido no Parecer como R$8-12k. Apresente isso e pergunte se quer ajustar."

Tool Use como agente — O agente decide sozinho quando chamar ferramentas (buscar benchmark, gerar perguntas WSI, salvar rascunho), não porque o código mandou.

O que temos hoje (a realidade)
Olhando o código, o que chamamos de "agentes" na LIA são na verdade funções que chamam LLM com prompts específicos. Não são agentes autônomos. Veja:

JobWizardGraph — é uma pipeline linear (A→B→C→D), não um agente que raciocina
9 "domínios de agentes" (Sourcing, Job Management, CV Screening, etc.) — são módulos de código com prompts, não agentes que decidem o que fazer
Intent classification — regex com fallback, não compreensão real
Orquestração — o FastRouter roteia para o domínio certo, mas cada domínio executa um script fixo
"Multi-agent" — na prática é chamada sequencial de funções, não agentes que colaboram
É como ter um call center onde cada atendente segue um script. Funciona, mas não é inteligente.

O que seria um sistema de agentes de verdade
Sistemas como Manus AI, CrewAI, AutoGen, ou mesmo o que empresas como Cognition (Devin) constroem, têm 5 pilares que a LIA não tem:

1. Agentes com autonomia — Cada agente tem um objetivo, ferramentas, e decide sozinho como alcançar. Não segue um script.

2. Loop de raciocínio (ReAct) — Pensar → Agir → Observar → Pensar de novo. Não pipeline linear.

3. Memória rica — Working memory (o que estou fazendo agora), episodic memory (o que aconteceu antes), semantic memory (o que sei sobre o mundo).

4. Comunicação entre agentes — Agentes delegam tarefas entre si, pedem ajuda, compartilham contexto.

5. Planejamento — Dado um objetivo complexo, o agente cria um plano, executa, adapta quando algo muda.

Por que não temos isso?
Porque a arquitetura foi construída incrementalmente — cada feature foi adicionada como um módulo isolado com seu próprio prompt. O resultado é um sistema que tem muitas capacidades mas nenhuma inteligência unificada. É um monolito de prompts disfarçado de multi-agent.

O que propor
A transformação real seria repensar a arquitetura inteira para ser agent-first. Não é refatorar o wizard — é refatorar a LIA como produto. Isso envolveria:

Agent Core — Um loop de raciocínio central (ReAct) que recebe qualquer input e decide o que fazer
Specialized Skills (não "agentes") — Cada domínio vira uma skill que o agente central pode invocar
Unified Memory — Um sistema de memória que persiste contexto, preferências, histórico de decisões
Tool Registry — Todas as ações possíveis registradas como tools que o agente decide quando usar
Planning Layer — Dado um objetivo ("criar vaga de Dev Senior"), o agente cria e executa um plano
Orchestrator real — Ao invés do FastRouter fazer routing por regex, o agente central entende e delega
Isso é uma mudança de paradigma, não uma refatoração. Mas é o que diferenciaria a LIA de "plataforma com IA" para "plataforma de agentes de IA".

quais sao os riscos disso?

Boa pergunta. Vou ser direto sobre os riscos reais:

Riscos Altos
1. Latência e custo
Hoje cada mensagem faz 1-2 chamadas ao Claude. Com ReAct loop, pode fazer 3-5 chamadas por mensagem (pensar → agir → observar → pensar → responder). Isso significa:

Tempo de resposta de 2-3s pode ir para 5-10s
Custo de API multiplica por 2-3x
Experiência do usuário pode sofrer se a resposta demora
2. Imprevisibilidade
Agentes autônomos às vezes fazem coisas inesperadas. Se o agente decide sozinho que precisa chamar uma tool, pode executar algo que o recrutador não pediu. Quanto mais autonomia, mais risco de comportamento indesejado. Isso requer guardrails sólidos.

3. Regressão do que funciona
Hoje o sistema funciona de forma previsível. Cada etapa faz o que foi programado. Ao trocar para decisões do agente, coisas que funcionavam podem quebrar porque o LLM decidiu diferente. O risco de "estava funcionando ontem e hoje não" é real.

4. Debugging mais difícil
Com código fixo, quando algo dá errado, você olha o código e acha o problema. Com agente autônomo, o "bug" pode ser uma decisão do LLM que não tem log claro. Precisa de observabilidade robusta (logging de cada passo de raciocínio).

Riscos Médios
5. Complexidade de manutenção
Adicionar uma camada de agente sobre os domínios cria mais uma coisa para manter. Se o prompt do agente central não for bem escrito, ele pode rotear mal, ignorar domínios, ou gerar respostas genéricas.

6. Dependência do modelo
O comportamento do agente depende 100% do Claude. Se a Anthropic muda o modelo ou tem instabilidade, o agente pode mudar de comportamento de um dia para o outro. Hoje, com código fixo, isso afeta menos.

7. Escopo pode crescer demais
Refatorar "só a orquestração" pode virar refatorar tudo. É fácil entrar num ciclo de "preciso mudar isso também para funcionar direito".

Riscos Baixos (mitigáveis)
8. Perda de dados — Baixo, porque os domínios/banco não mudam
9. Frontend quebrar — Baixo, se a interface aceitar respostas dinâmicas do backend

Como mitigar
Abordagem incremental — Não trocar tudo de uma vez. Começar pelo wizard (1 chat), validar, depois expandir
Fallback — Se o agente falha ou demora, cair de volta para o comportamento fixo atual
Guardrails — Lista de ações que o agente NUNCA pode fazer sozinho (deletar vaga, publicar sem confirmação)
Cache de raciocínio — Para perguntas comuns, cachear a decisão do agente
Feature flag — Manter os dois sistemas (fixo e agente) e alternar com flag
Observabilidade — Log de cada passo do ReAct para debugging
Resumindo: os riscos existem, mas são gerenciáveis se a implementação for incremental (começar pelo wizard), com fallback (se falhar, volta ao comportamento atual), e com feature flag (liga/desliga). O maior risco é latência/custo, que precisa ser medido e otimizado.
