🔍 ANÁLISE PROFUNDA: Por Que Estes Problemas Estão Acontecendo
1. CAUSA RAIZ DOS PROBLEMAS
Problema Principal: Evolução Desacoplada
O sistema foi construído em camadas separadas que evoluíram de forma independente:

Frontend (Next.js/React)     →    Backend (FastAPI)     →    Agentes (LangGraph)
     wizard-config.ts              orchestrator.py            nodes.py
         ↓                              ↓                         ↓
  7 stages definidas            Mapeamentos parciais       6 stages no enum
  job-summary existe            job-summary ignorada       job-summary não existe
Por que aconteceu?

O frontend foi desenhado com UX em mente (job-summary = etapa de proposta)
O backend foi implementado seguindo os stages que existiam no enum
O enum do backend foi baseado em um fluxo mais simples (sem etapa de "proposta")
Ninguém atualizou todos os pontos quando novos stages foram adicionados
Problema 2: Sistema de Confirmação "Soft"
# O que existe:
STAGES_REQUIRING_CONFIRMATION = ["initial", "title_department", ...]
# O que o código faz:
if requires_confirmation:
    if intent == WizardIntent.CONFIRM.value:
        should_advance = True  # ✅ Só avança se confirmar
    else:
        # Mas NÃO bloqueia explicitamente!
        pass  # ← Problema: pode cair no else abaixo
else:
    if can_auto_advance:  # ← Isso pode ser True
        should_advance = True  # ← E avançar automaticamente!
Por que aconteceu?

A lógica foi construída para "permitir" avanço em casos específicos
Não há um "bloqueio duro" que diga "NÃO AVANÇA SEM CONFIRMAÇÃO"
O should_auto_advance() pode retornar True se os campos estiverem preenchidos
Problema 3: Nomes de Campos Divergentes
Onde	Nome do Campo	Por que
LLM Extraction	required_skills	Prompt pede "required_skills"
Backend State	required_skills	Copia do LLM
Frontend	technical_skills	UI usa termo diferente
Job Draft	required_skills	Não há transformação
Por que aconteceu?

O backend foi desenhado primeiro com terminologia de "requisitos"
O frontend foi desenhado com UX focada no usuário ("técnicas" é mais claro)
Não há camada de normalização entre os dois
Problema 4: Response Types Apenas Sugestivos
# O prompt diz:
"## TIPOS DE RESPOSTA (OBRIGATÓRIO usar um destes)
### 1. ASK_CLARIFY..."
# Mas o código:
response = await self.llm.generate(prompt=..., provider="gemini")
state["response_text"] = response.strip()  # ← Apenas texto livre!
# Não há parsing de tipo, não há validação, não há estrutura forçada
Por que aconteceu?

A implementação inicial focou em "funcionar"
Structured Output do LLM não foi implementado
O LLM pode seguir ou não as instruções (não há garantia)
2. O QUE ESPERAR DAS MELHORIAS
Antes vs Depois - Comparação Prática
Situação	ANTES	DEPOIS
Usuário diz "quero criar vaga de dev"	LIA extrai, pode avançar automaticamente	LIA extrai, MOSTRA resumo, ESPERA confirmação
Stage "job-summary" no frontend	Erro ou comportamento indefinido	Stage reconhecida, prompt especializado
Extração de "Python e SQL"	Salvo como required_skills	Salvo como technical_skills (normalizado)
Confirmação em salary	Pode pular se confiança alta	OBRIGATÓRIO confirmar, bloqueio duro
Início: "usar template"	Passa por field_extractor	Vai direto para response_generator
Métricas de Melhoria Esperadas
Confiabilidade da Jornada: 60% → 95%

Stages corretas, transições previsíveis
Consistência de Dados: 70% → 98%

Campos normalizados, sem perdas
Experiência Conversacional: Adequada → Excelente

Confirmações claras, resumos estruturados
Respostas Estruturadas: 0% → 100%

Tipos enforçados, formato previsível
