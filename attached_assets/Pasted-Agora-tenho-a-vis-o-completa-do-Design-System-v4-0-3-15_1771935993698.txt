Agora tenho a visão completa do Design System v4.0 (3.155 linhas, 5 partes). Vou entregar a análise completa como especialista em design e desenvolvimento front-end.

Aqui está minha Análise Especializada do Design System v4.0 da WeDo Talent:

Visão Geral
O DS v4.0 é um documento robusto e bem estruturado com 5 partes: Fundamentos, Componentes, Padrões, Implementação e Catálogos. A filosofia "90% monocromático + 10% cores WeDo" é sólida e alinhada com tendências atuais (ElevenLabs, Linear, Vercel). No entanto, identifiquei incoerências internas, lacunas e oportunidades de melhoria.

PONTOS FORTES
Filosofia visual coerente — A regra 90/10 é excelente para plataformas B2B/SaaS
Sistema de tokens duplo (CSS + TypeScript) — Permite uso tanto em CSS puro quanto em componentes React
Catálogo de modais — 58+ modais catalogados por tamanho é raro e muito útil
Acessibilidade documentada — WCAG 2.1 AA com tabela de contraste
Sistema de motion — 3 estados (Static/Resting/Reactive) bem definidos
Mapeamento Vuetify — Preparação para migração Vue.js já documentada
Brain Icon como identidade — Uso estratégico bem delimitado com "onde usar" e "onde não usar"
INCOERÊNCIAS IDENTIFICADAS
1. Tipografia — Proporções inconsistentes entre documentação e changelog

Seção 1.4 diz: "Open Sans (85%) + Inter (10%) + Source Serif 4 (5%)"
Changelog diz: "Open Sans (85%) + Inter (10%) + Source Serif 4 (5%)" — OK aqui
Mas o replit.md diz: "Open Sans 60% + Inter 40%" — conflito direto
Recomendação: Unificar. A proporção do DS v4.0 (85/10/5) parece mais adequada para UI.
2. Border Radius — Conflito entre seção 1.8 e replit.md

DS v4.0 seção 1.8: Cards = rounded-lg (12px), Modais = rounded-lg (12px), Botões = rounded-md (8px)
replit.md diz: "rounded-md (8px) para cards/modais, rounded (6px) para botões"
Os valores CSS nos componentes usam border-radius: 12px para cards e 6px para botões
Incoerência: A tabela 1.8.1 define rounded como 6px e rounded-md como 8px, mas os componentes CSS usam 12px para cards (que seria rounded-lg)
3. Tamanho mínimo de texto — Contradição prática

Seção 1.4 proíbe textos abaixo de 10px
Mas lia-badge usa font-size: 0.625rem = exatamente 10px
Headers de tabela usam 11px — OK
Risco: Em dispositivos com DPI baixo, 10px pode ficar ilegível
4. Botão Ghost — font-weight inconsistente

Primary, Secondary, Outline, Destructive: todos font-weight: 600
Ghost: font-weight: 500
Menor mas cria inconsistência visual quando botões aparecem lado a lado
5. Card radius vs Modal radius

Cards CSS: border-radius: 12px (rounded-lg)
Modais na estrutura (seção 2.4.3): rounded-lg
Mas a tabela 1.8.2 mostra Modais como rounded-lg e Cards como rounded-lg — OK neste ponto, mas conflita com o replit.md que diz rounded-md (8px)
6. Sombras — Inconsistência entre tabela e componentes

Tabela 1.7.1: shadow-sm = 0 1px 2px rgba(0,0,0,0.05)
Card Elevated CSS: box-shadow: 0 4px 6px rgba(0,0,0,0.05) — parece shadow-md mas com 0.05 ao invés de 0.1
Tabela diz shadow-md = 0 4px 6px rgba(0,0,0,0.1) mas o card elevated usa 0.05
7. Modal sizes — Proibição de max-w-xl mas sem justificativa técnica

Proíbe max-w-xl (576px), max-w-3xl (768px), max-w-5xl (1024px)
Mas não explica por quê. O gap entre S (448px) e M (512px) é apenas 64px, enquanto entre L (672px) e XL (896px) é 224px — distribuição desigual
LACUNAS IMPORTANTES
1. Sem Dark Mode nos componentes

Seção 4.3 menciona classes .dark para texto, mas nenhum componente (buttons, cards, inputs, modais, toasts) tem especificação dark mode
Brain Icon diz "mantém #60BED1 em dark mode" mas o restante fica indefinido
2. Sem sistema de spacing documentado

Não há escala de spacing definida (4px base? 8px? t-shirt sizes?)
Os componentes usam valores Tailwind arbitrários (p-4, p-6, gap-2, gap-3) sem regra clara
3. Sem tokens de z-index

Tooltip e Popover mencionam z-index: 50 mas não há escala completa
Modais, overlays, toasts, dropdowns competem por z-index sem sistema definido
4. Sem sistema de grid/layout

Layout section (1.5) mostra diagrama ASCII mas sem sistema de grid documentado
Não define container widths, padding interno, ou gutters
5. Sem estados de componentes compostos

Dropdown select: especifica aparência mas não estados (open, searching, no results, loading)
Combobox/Autocomplete: ausente do DS
Date picker: ausente
File upload: ausente
6. Sem padrão para dados em tempo real

A plataforma tem chat, pipeline kanban, notificações — tudo real-time
Não há padrão para indicadores de "ao vivo", presença, typing indicators
7. Sem design tokens para o chat (interface principal)

O chat é declarado como "interface principal" mas o DS não tem seção dedicada
Bolhas de mensagem, áreas de input, indicadores de typing, layouts de conversa — tudo ausente

ESTRUTURA: Está boa, mas incompleta para o propósito
A divisão em 5 partes faz sentido:

Parte 1 (Fundamentos) — serve UX/UI
Parte 2 (Componentes) — serve ambos
Parte 3 (Padrões) — serve ambos
Parte 4 (Implementação) — serve devs
Parte 5 (Catálogos) — serve ambos
Problema estrutural principal: o documento mistura decisões de design com código de implementação sem separação clara. Um designer não precisa ver CSS, e um dev React não precisa saber a justificativa filosófica do motion. Idealmente seriam camadas separadas, mas como é um documento único, precisa de marcações claras de audiência.

O QUE ESTÁ CORRETO
Filosofia 90/10 — decisão sólida, alinhada com mercado (Linear, Vercel, ElevenLabs)
Sistema de 3 fontes com propósito — cada fonte tem domínio claro
Hierarquia de botões — 6 variantes com estados completos
Brain Icon — identidade visual bem documentada com regras de uso e anti-padrões
Tabela de migração v3 → v4 — muito útil para equipe existente
Mapeamento Tailwind ↔ Vuetify — essencial para a migração futura
Catálogo de modais por tamanho — raro e muito prático
Checklist de padronização — excelente para code review
O QUE ESTÁ INCORRETO OU INCONSISTENTE
1. Border-radius — 3 verdades diferentes

Tabela 1.8.2: Cards = rounded-lg (12px), Buttons = rounded-md (8px)
CSS dos componentes: Cards = 12px, Buttons = 6px
replit.md: Cards/Modais = rounded-md (8px), Buttons = rounded (6px)
Precisa unificar em UMA tabela canônica
2. Card Elevated — sombra errada

Tabela 1.7.2: Card hover = shadow-md = 0 4px 6px rgba(0,0,0,0.1)
CSS .lia-card-elevated: box-shadow: 0 4px 6px rgba(0,0,0,0.05) — opacidade diferente
Qual é o correto? 0.05 ou 0.1?
3. Touch target — abaixo do mínimo WCAG

Seção 3.6.1 diz: "Touch Target 44x44px mínimo" e "Botões h-9 (36px) + padding"
Mas botão XS tem height 28px e SM tem 32px — ambos abaixo de 44px
A justificativa "+ padding" é fraca. WCAG 2.5.8 (AAA) pede 44x44, e WCAG 2.5.5 (AA) pede no mínimo 24x24 de target size. Precisa esclarecer que XS/SM são para uso desktop-only
4. Proporção tipográfica no replit.md vs DS

DS: Open Sans 85% + Inter 10% + Source Serif 4 5%
replit.md: Open Sans 60% + Inter 40%
São documentos que o time consulta — precisam falar a mesma língua
5. Input focus — ring inconsistente

CSS do input: box-shadow: 0 0 0 3px rgba(17, 24, 39, 0.1) — opacidade 0.1
Focus ring padrão (seção 3.1.2): box-shadow: 0 0 0 3px rgba(17, 24, 39, 0.2) — opacidade 0.2
Input tem metade da visibilidade dos outros componentes

Slots e composição Vuetify	Como traduzir a estrutura de Card, Modal, etc. para slots Vuetify
