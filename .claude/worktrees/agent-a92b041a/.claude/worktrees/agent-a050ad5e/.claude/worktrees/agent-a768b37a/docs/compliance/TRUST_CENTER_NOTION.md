## 2.1 🔵 SOC 2 Type II

> 💡 **O que é SOC 2?**
>
> SOC 2 (Service Organization Control 2) é uma certificação de segurança criada pelo AICPA (Instituto Americano de Contadores Públicos Certificados). É considerado o **padrão ouro global** de segurança para empresas de software e SaaS. A certificação comprova que uma empresa protege adequadamente os dados de seus clientes através de controles rigorosos auditados por terceiros independentes.

### Informações da Certificação

| Campo | Valor |
|-------|-------|
| 🚦 **Status** | 🔄 Em Progresso |
| 📅 **Previsão de Conclusão** | Junho 2026 |
| 🏢 **Plataforma de Compliance** | Vanta/Drata |
| 📋 **Escopo** | Segurança, Disponibilidade, Integridade de Processamento, Confidencialidade |
| ⏱️ **Período de Observação** | 6-12 meses de monitoramento contínuo |
| 🔄 **Renovação** | Anual (reauditoria a cada 12 meses) |

### 2.1.1 ❓ O que é SOC 2 e para que serve?

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Uma auditoria independente que verifica se uma empresa tem controles adequados para proteger dados de clientes |
| 🎯 **Para que serve?** | Comprovar para clientes e parceiros que a segurança não é só promessa - é prática verificada |
| 🔍 **Quem audita?** | Empresas de auditoria independentes e certificadas (como EY, Deloitte, KPMG, ou firmas especializadas) |
| 📊 **O que é avaliado?** | Políticas, procedimentos, controles técnicos e evidências de funcionamento durante meses |
| 💼 **Quem exige?** | Grandes empresas, bancos, fintechs, empresas de saúde e qualquer organização que leva segurança a sério |

### 2.1.2 🔍 Type I vs Type II - Qual a diferença?

| 📊 Aspecto | 📸 Type I | 🎬 Type II |
|-----------|----------|-----------|
| **Analogia** | Foto de um momento | Filme de 6-12 meses |
| **O que verifica** | Se os controles existem em uma data específica | Se os controles funcionam consistentemente ao longo do tempo |
| **Duração da auditoria** | Avaliação pontual (1-2 meses) | Monitoramento contínuo (6-12 meses) |
| **Nível de confiança** | Bom | **Excelente** (mais rigoroso) |
| **Preferência do mercado** | Aceitável para empresas novas | **Exigido por grandes clientes** |

> 💡 **Por que buscamos Type II?** Porque é o padrão exigido por empresas enterprise. Type I prova que você TEM controles; Type II prova que eles FUNCIONAM ao longo do tempo.

### 2.1.3 📋 Os 5 Trust Service Criteria (TSC)

O SOC 2 avalia empresas em até 5 critérios. A WeDo Talent está sendo auditada em 4 deles:

| # | 🏆 Critério | 📋 O que avalia | 🎯 Nosso escopo |
|---|-----------|----------------|----------------|
| 1 | 🔐 **Segurança** (obrigatório) | Proteção contra acesso não autorizado | ✅ Incluído |
| 2 | 🟢 **Disponibilidade** | Sistema acessível conforme acordado (uptime) | ✅ Incluído |
| 3 | ⚙️ **Integridade de Processamento** | Processamento preciso e sem erros | ✅ Incluído |
| 4 | 🔒 **Confidencialidade** | Proteção de informações confidenciais | ✅ Incluído |
| 5 | 👤 **Privacidade** | Tratamento de dados pessoais | 🔄 Coberto pela LGPD |

**Detalhamento dos critérios:**

| 🏆 Critério | 📋 O que significa na prática | 🛡️ Como atendemos |
|------------|------------------------------|------------------|
| 🔐 **Segurança** | Firewalls, criptografia, controle de acesso, detecção de intrusão | AES-256, TLS 1.3, MFA, RBAC, monitoramento 24/7 |
| 🟢 **Disponibilidade** | Sistema funcionando quando prometido, backups, disaster recovery | SLA 99.9%, backups diários, DR testado trimestralmente |
| ⚙️ **Integridade** | Dados processados corretamente, sem alterações não autorizadas | Logs imutáveis, validação de entrada, checksums |
| 🔒 **Confidencialidade** | Dados sensíveis protegidos de divulgação indevida | Classificação de dados, criptografia, least privilege |

## 2.2 🟢 ISO 27001:2022

> 💡 **O que é ISO 27001?**
>
> ISO 27001 é uma norma internacional que especifica os requisitos para um **Sistema de Gestão de Segurança da Informação (SGSI)**. Diferente de controles pontuais, a ISO 27001 exige que a empresa tenha um SISTEMA completo e contínuo de gestão de segurança, com processos documentados, responsabilidades definidas e melhoria constante.

### Informações da Certificação

| Campo | Valor |
|-------|-------|
| 🚦 **Status** | ⏳ Planejado |
| 📅 **Previsão de Certificação** | Dezembro 2026 |
| 📋 **Versão** | ISO 27001:2022 (versão mais recente) |
| 📋 **Escopo** | Sistema de Gestão de Segurança da Informação (SGSI) |
| 🏢 **Organismos Certificadores** | BSI, Bureau Veritas, DNV, SGS, entre outros |
| 🔄 **Validade** | 3 anos (com auditorias de manutenção anuais) |

### 2.2.1 ❓ O que é um SGSI (Sistema de Gestão de Segurança da Informação)?

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Um conjunto integrado de políticas, processos, procedimentos, controles e responsabilidades para gerenciar a segurança da informação |
| 🎯 **Para que serve?** | Garantir que a segurança não dependa de pessoas específicas, mas seja um SISTEMA que funciona independentemente de quem está operando |
| 🔄 **Ciclo PDCA** | Plan (Planejar) → Do (Executar) → Check (Verificar) → Act (Agir) - melhoria contínua |
| 📋 **Documentação** | Políticas escritas, procedimentos documentados, registros de evidências |
| 👥 **Responsabilidades** | Papéis e responsabilidades claramente definidos para segurança |

### 2.2.2 📊 Diferença entre SOC 2 e ISO 27001

| 📊 Aspecto | 🔵 SOC 2 | 🟢 ISO 27001 |
|-----------|---------|-------------|
| **Origem** | EUA (AICPA) | Internacional (ISO) |
| **Foco** | Controles operacionais funcionando | Sistema de gestão completo |
| **Escopo** | 5 Trust Service Criteria | 114 controles em 14 domínios |
| **Resultado** | Relatório de auditoria | Certificado oficial |
| **Renovação** | Anual | 3 anos (com auditorias anuais) |
| **Preferência** | EUA, Canadá, startups | Europa, Ásia, enterprises |
| **Analogia** | "Seus controles funcionam?" | "Você tem um SISTEMA de segurança?" |

> 💡 **Por que buscamos os dois?** Porque atendem públicos diferentes. Clientes americanos pedem SOC 2; clientes europeus e grandes corporações pedem ISO 27001. Ter ambos demonstra maturidade máxima.

### 2.2.3 📋 Os 14 Domínios da ISO 27001

A ISO 27001:2022 organiza 114 controles de segurança em 14 domínios:

| # | 📋 Domínio | 📝 O que cobre |
|---|-----------|---------------|

| A.6 | Organização da Segurança | Papéis, responsabilidades, dispositivos móveis |
| A.7 | Segurança em RH | Contratação, treinamento, desligamento |
| A.8 | Gestão de Ativos | Inventário, classificação, manuseio de mídia |
| A.9 | Controle de Acesso | Autenticação, autorização, senhas |
| A.10 | Criptografia | Políticas e gestão de chaves |
| A.11 | Segurança Física | Data centers, equipamentos, descarte |
| A.12 | Segurança Operacional | Backup, logs, vulnerabilidades, malware |
| A.13 | Segurança de Comunicações | Redes, transferência de informações |
| A.14 | Aquisição e Desenvolvimento | Segurança em sistemas e software |
| A.15 | Relacionamento com Fornecedores | Gestão de terceiros |
| A.16 | Gestão de Incidentes | Resposta, reporte, lições aprendidas |
| A.17 | Continuidade de Negócios | Disaster recovery, resiliência |
| A.18 | Conformidade | Leis, regulamentos, auditorias |

### 2.2.4 🎯 Por que a WeDo Talent busca ISO 27001?

## 2.4 🤖 AI Bias Audit (NYC LL144)

> 💡 **O que é Auditoria de Viés em IA?**
>
> É uma avaliação independente que verifica se algoritmos de Inteligência Artificial tratam todos os grupos de pessoas de forma justa, sem discriminar por características como gênero, raça, idade ou outras características protegidas. É especialmente crítico para IA usada em recrutamento, onde vieses podem excluir injustamente candidatos qualificados.

### 2.4.1 ❓ O que é a NYC Local Law 144?

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Uma lei de Nova York que exige auditoria anual de viés em ferramentas automatizadas de emprego (AEDT) |
| 📅 **Vigente desde** | Janeiro 2023 |
| 🎯 **Para que serve?** | Garantir que IA usada em recrutamento não discrimine candidatos |
| 📊 **O que avalia?** | Taxa de seleção por gênero, raça/etnia e interseções |

| 🎯 Motivo | 📋 Explicação |
|----------|--------------|

| 🔮 **Antecipação regulatória** | A tendência global é exigir auditorias - melhor estar preparado |

### 2.4.3 📊 O que uma auditoria de viés avalia?

| 📊 Métrica | 📋 O que mede | 🎯 Threshold aceitável |
|-----------|-------------|----------------------|
| **Impact Ratio (Gênero)** | Taxa de seleção de mulheres vs homens | ≥ 0.80 (regra 4/5 da EEOC) |
| **Impact Ratio (Raça/Etnia)** | Taxa de seleção de minorias vs maioria | ≥ 0.80 (regra 4/5 da EEOC) |
| **Interseccionalidade** | Combinações (ex: mulheres negras vs homens brancos) | Análise de subgrupos |
| **Consistência** | Resultados similares em diferentes períodos | Variação aceitável |

> 💡 **O que é a "Regra 4/5"?** Se um grupo é selecionado em taxa menor que 80% (4/5) da taxa do grupo mais selecionado, há evidência de impacto adverso. Exemplo: se 50% dos homens passam, pelo menos 40% das mulheres devem passar.

> 💡 **O que é a LGPD?**
>
> A LGPD (Lei Geral de Proteção de Dados - Lei 13.709/2018) é a principal lei brasileira sobre privacidade e proteção de dados pessoais. Inspirada no GDPR europeu, ela estabelece regras claras sobre como empresas podem coletar, armazenar, processar e compartilhar informações de pessoas físicas no Brasil.

| 7 | **Segurança** | Medidas técnicas para proteger dados | Criptografia AES-256, MFA, controles de acesso |
| 8 | **Prevenção** | Adotar medidas para evitar danos | Análise de riscos e auditorias regulares |
| 9 | **Não Discriminação** | Dados não podem ser usados para discriminar | IA auditada contra viés, campos protegidos excluídos |

## 3.3 🗽 NYC LL144 - Lei de Auditoria de Viés em IA de Nova York

> 💡 **O que é NYC LL144?**
>
> A NYC Local Law 144 (Lei Local 144 de Nova York) é a primeira lei do mundo a exigir auditorias anuais de viés em ferramentas automatizadas de emprego (AEDT - Automated Employment Decision Tools). Vigente desde janeiro de 2023, ela obriga empresas que usam IA em recrutamento a publicar resultados de auditoria que mostrem se a ferramenta discrimina por gênero, raça ou etnia.

### Informações do Framework

| Campo | Valor |
|-------|-------|
| 🚦 **Status** | ⏳ Planejado (Auditoria prevista para Set 2026) |
| 📅 **Vigente desde** | Janeiro 2023 |
| 🏛️ **Órgão Regulador** | DCWP (Department of Consumer and Worker Protection - NYC) |
| 💰 **Penalidades** | Até $1.500 por violação (por candidato afetado) |
| 🔗 **Link Oficial** | [nyc.gov](https://www.nyc.gov/site/dcwp/workers/automated-employment-decision-tools.page) |

### 3.3.1 ❓ O que é e para que serve?

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Uma lei que exige auditoria anual de viés em qualquer ferramenta de IA usada para triagem de candidatos ou promoções |
| 🎯 **Para que serve?** | Garantir que algoritmos de recrutamento não discriminem candidatos por características protegidas (gênero, raça, etnia) |
| 📊 **O que é auditado?** | Taxas de seleção por grupo demográfico - verifica-se se algum grupo é sistematicamente prejudicado |
| 📢 **Transparência** | Resultados da auditoria devem ser publicados no site da empresa |
| 👤 **Notificação** | Candidatos devem ser informados de que IA está sendo usada em sua avaliação |

### 3.3.2 📊 O que a auditoria de viés avalia?

| 📊 Métrica | 📋 O que mede | 🎯 Threshold aceitável |
|-----------|-------------|----------------------|
| **Impact Ratio (Gênero)** | Taxa de seleção de mulheres ÷ taxa de seleção de homens | ≥ 0.80 (regra 4/5 da EEOC) |
| **Impact Ratio (Raça/Etnia)** | Taxa de seleção de cada grupo ÷ taxa do grupo mais selecionado | ≥ 0.80 para cada grupo |
| **Interseccionalidade** | Análise de combinações (ex: mulheres negras vs homens brancos) | Variações aceitáveis |
| **Consistência** | Resultados similares em diferentes períodos e datasets | Estabilidade comprovada |

> 💡 **O que é a "Regra 4/5"?** 
>
> A regra 4/5 (ou regra dos 80%) é um padrão da EEOC (Equal Employment Opportunity Commission): se um grupo é selecionado em taxa menor que 80% da taxa do grupo mais selecionado, há evidência estatística de impacto adverso (discriminação). 
>
> **Exemplo:** Se 50% dos homens são selecionados, pelo menos 40% das mulheres devem ser selecionadas (50% × 80% = 40%). Se apenas 30% das mulheres passam, há evidência de viés.

| 🎯 Motivo | 📋 Explicação |
|----------|--------------|
| 🌍 **Referência global** | NYC LL144 é a lei MAIS DETALHADA do mundo sobre auditoria de viés em IA de recrutamento |
| 🇧🇷 **Antecipação regulatória** | Brasil está desenvolvendo legislação similar (PL 2338/2023) - melhor estar preparado |
| 🏢 **Clientes multinacionais** | Empresas com operações nos EUA precisam de fornecedores auditados |
| 📋 **Melhores práticas** | Seguir o padrão mais rigoroso demonstra compromisso genuíno com equidade |
| ⚖️ **Proteção legal** | Auditoria documenta boa-fé em caso de questionamentos sobre discriminação |

### 3.3.4 🔬 Como funciona a auditoria na prática?

| # | 🔄 Etapa | 📋 O que acontece |
|---|---------|------------------|
| 1 | 📊 **Coleta de dados** | Auditor independente recebe dados anonimizados de candidatos e decisões da IA |
| 2 | 📈 **Análise estatística** | Cálculo de taxas de seleção por gênero, raça/etnia e interseções |
| 3 | 🔍 **Identificação de viés** | Verificação se algum grupo é selecionado em taxa menor que 80% do grupo mais selecionado |
| 4 | 📋 **Relatório** | Documento formal com metodologia, resultados, impactos e recomendações |
| 5 | 📢 **Publicação** | Resultados são publicados no site (transparência obrigatória) |
| 6 | 🔧 **Remediação** | Se houver viés detectado, correções são implementadas e nova auditoria é realizada |

### 3.3.5 🛡️ Proteções já implementadas na LIA

Enquanto aguardamos a auditoria formal, a LIA já possui diversas proteções contra viés:

| 🛡️ Proteção | 📋 Descrição | 🔍 Como funciona |

### 3.3.6 🎯 Como afeta candidatos e clientes?

| 👤 Para quem? | ✅ Benefício | 📋 Exemplo prático |
|--------------|-------------|-------------------|
| 🧑‍💼 **Candidatas (mulheres)** | Garantia de não discriminação por gênero | Auditoria verifica se mulheres são tratadas igualmente |

## 3.4 🇪🇺 EU AI Act - Regulamento Europeu de Inteligência Artificial

>

Aprovado em 2024, ele classifica sistemas de IA por nível de risco e estabelece requisitos proporcionais. **IA usada em recrutamento é classificada como ALTO RISCO**, exigindo requisitos rigorosos de transparência, supervisão humana e gestão de riscos.

| 💰 **Penalidades** | Até €35 milhões ou 7% do faturamento global |
| 🔗 **Link Oficial** | [eur-lex.europa.eu](https://eur-lex.europa.eu/legal-content/PT/TXT/?uri=CELEX:32024R1689) |

### 3.4.1 ❓ O que é e para que serve?

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|

| 🚦 Nível de Risco | 📋 Exemplos | 📝 Requisitos |

## 3.5 🌴 CA FEHA - California Fair Employment and Housing Act

> 💡 **O que é CA FEHA?**
>
> O FEHA (Fair Employment and Housing Act) é a principal lei antidiscriminação trabalhista da Califórnia, EUA. É considerada uma das mais rigorosas do mundo, proibindo discriminação em todas as etapas do emprego, incluindo recrutamento. Recentemente, a Califórnia tem aplicado o FEHA a algoritmos de recrutamento que possam ter impacto discriminatório.

### Informações do Framework

| Campo | Valor |
|-------|-------|
| 🚦 **Status** | 👀 Monitorando |
| 📅 **Vigente desde** | 1959 (atualizações constantes) |
| 🏛️ **Órgão Regulador** | CRD (Civil Rights Department - California) |
| 💰 **Penalidades** | Sem limite máximo - danos compensatórios e punitivos |
| 🔗 **Link Oficial** | [ca.gov](https://calcivilrights.ca.gov/employment/) |

### 3.5.1 ❓ O que é e para que serve?

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | A lei estadual da Califórnia que proíbe discriminação em emprego, moradia e acomodações públicas |
| 🎯 **Para que serve?** | Garantir igualdade de oportunidades no trabalho, independente de características protegidas |
| 📋 **Características protegidas** | Raça, cor, religião, sexo, gênero, orientação sexual, idade, deficiência, nacionalidade, e mais |
| 🤖 **Aplicação a IA** | Desde 2020, reguladores californianos têm aplicado o FEHA a algoritmos de recrutamento |
| 🌍 **Influência global** | A Califórnia frequentemente lidera em legislação trabalhista, influenciando outros estados e países |

### 3.5.2 📋 Classes Protegidas pelo FEHA

O FEHA protege contra discriminação baseada em:

| 🛡️ Característica | 📋 Exemplos de proteção |
|------------------|------------------------|
| 👤 **Raça/Cor** | Não rejeitar candidatos por cor da pele ou etnia |
| 🙏 **Religião** | Não excluir por crenças religiosas ou falta delas |
| ⚧️ **Sexo/Gênero** | Igualdade entre homens, mulheres, não-binários |
| 🏳️‍🌈 **Orientação Sexual** | Não discriminar por orientação sexual |
| 🎂 **Idade (40+)** | Não excluir candidatos por serem "velhos demais" |
| ♿ **Deficiência** | Acomodações razoáveis para pessoas com deficiência |
| 🌍 **Nacionalidade** | Não rejeitar por país de origem ou sotaque |
| 🤰 **Gravidez** | Não discriminar candidatas grávidas |
| 🎖️ **Veteranos** | Proteções especiais para veteranos militares |

### 3.5.3 📊 Como o FEHA se aplica a IA de Recrutamento?

| 📋 Aspecto | 📝 Aplicação a IA |
|-----------|------------------|
| **Discriminação direta** | IA que usa explicitamente características protegidas viola o FEHA |
| **Discriminação indireta (impacto adverso)** | IA com resultados que prejudicam desproporcionalmente um grupo pode violar o FEHA, mesmo sem intenção |
| **Responsabilidade do empregador** | A empresa é responsável por discriminação causada por ferramentas de terceiros que ela usa |
| **Acomodações** | IA deve permitir acomodações para candidatos com deficiência |
| **Transparência** | Candidatos têm direito de saber se IA está sendo usada em sua avaliação |

## 4.2 🔒 Criptografia

> 💡 **O que é criptografia?**
>
> Criptografia é como um cofre digital: transforma dados legíveis em códigos incompreensíveis para quem não tem a "chave". Mesmo que alguém intercepte os dados, não conseguirá entendê-los sem a chave correta. Usamos criptografia em TODAS as camadas da plataforma.

### Resumo dos Padrões

| 🔐 Tipo | 🛡️ Padrão | 📋 Descrição |
|--------|----------|-------------|
| 💾 **Em Repouso** | AES-256 | Todos os dados armazenados são criptografados |
| 🌐 **Em Trânsito** | TLS 1.3 | Comunicação sempre criptografada |
| 🔑 **Senhas** | bcrypt + salt | Hash unidirecional com salt aleatório |
| 🗝️ **Chaves** | AWS KMS | Gerenciamento seguro de chaves |

---

### 4.2.1 💾 AES-256 (Dados em Repouso)

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | AES (Advanced Encryption Standard) com chave de 256 bits - o padrão de criptografia mais forte disponível comercialmente |
| 🔢 **O que significa 256?** | Refere-se ao tamanho da chave: 2^256 combinações possíveis. Para quebrar por força bruta, levaria bilhões de anos mesmo com os computadores mais potentes |
| 🏛️ **Quem usa?** | Governos, bancos, militares e as maiores empresas de tecnologia do mundo |
| 🎯 **Onde aplicamos?** | Banco de dados, backups, arquivos de currículos, gravações de entrevistas |
| 🛡️ **O que protege?** | Se alguém roubar um disco rígido ou acessar o servidor, os dados são ilegíveis sem a chave |

**Exemplo prático:** Imagine que seu currículo está guardado em um cofre com 2^256 combinações possíveis. Mesmo tentando 1 bilhão de combinações por segundo, levaria mais tempo que a idade do universo para abrir.

**Riscos minimizados:**
- ❌ Roubo de dados por invasão de servidores
- ❌ Vazamento por funcionários mal-intencionados
- ❌ Exposição de dados em backups perdidos ou roubados

---

### 4.2.2 🌐 TLS 1.3 (Dados em Trânsito)

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | TLS (Transport Layer Security) versão 1.3 - o protocolo que cria o "cadeado verde" no navegador |
| 🔄 **O que significa "em trânsito"?** | Dados enquanto viajam pela internet - do seu computador até nossos servidores |
| 🆕 **Por que 1.3?** | É a versão mais recente (2018), mais rápida e mais segura que versões anteriores |
| 🛡️ **O que protege?** | Impede que hackers interceptem dados enquanto você navega na plataforma |
| 🔍 **Como verificar?** | Olhe o cadeado 🔒 na barra de endereço do seu navegador |

**Exemplo prático:** Quando você faz login na plataforma, sua senha viaja pela internet como um código indecífrável. Mesmo que alguém intercepte a transmissão (como em um Wi-Fi público), verá apenas dados embaralhados.

**O que TLS 1.3 melhora sobre versões anteriores:**
| 📊 Aspecto | ⬆️ Melhoria |
|-----------|------------|
| ⚡ **Velocidade** | Conexão estabelecida em 1 ida-e-volta (vs 2 no TLS 1.2) |
| 🔐 **Algoritmos** | Remove algoritmos antigos e vulneráveis |
| 🛡️ **Forward Secrecy** | Mesmo que uma chave vaze no futuro, comunicações passadas permanecem seguras |

---

### 4.2.3 🔑 bcrypt + salt (Senhas)

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é bcrypt?** | Um algoritmo de hash especialmente projetado para senhas - lento de propósito para dificultar ataques |
| 🧂 **O que é salt?** | Um valor aleatório único adicionado a cada senha antes de criptografar, impedindo ataques de "rainbow table" |
| 🔄 **Hash vs Criptografia** | Hash é unidirecional - NÃO podemos recuperar sua senha, apenas verificar se a que você digitou está correta |
| 💡 **Por que "lento"?** | Se um hacker tentar milhões de senhas, cada tentativa demora, tornando ataques impraticáveis |
| 🛡️ **O que protege?** | Mesmo que alguém acesse nosso banco de dados, não conseguirá descobrir as senhas |

**Exemplo prático:** Sua senha "MinhaSenh@123" + salt aleatório "x7Kp9mQ2" → bcrypt → "$2b$12$LQv3c1yqBWVHxkd0LHAkCOYz6TtxMQJqhN..." (irreversível)

**Por que isso é melhor que outros métodos:**
| ❌ Método Inseguro | ✅ Nossa Abordagem |
|-------------------|-------------------|
| Armazenar senha em texto plano | Nunca armazenamos senhas legíveis |
| Usar MD5 ou SHA1 (rápidos) | bcrypt é propositalmente lento |
| Hash sem salt | Salt único para cada usuário |

### 4.2.4 🗝️ AWS KMS (Gerenciamento de Chaves)

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | AWS KMS (Key Management Service) - serviço gerenciado da Amazon para criar e controlar chaves de criptografia |
| 🎯 **Para que serve?** | Guardar as "chaves do cofre" de forma segura, separada dos dados criptografados |
| 🔐 **Proteção da chave** | Chaves ficam em hardware especializado (HSM) e nunca saem descriptografadas |
| 📋 **Auditoria** | Cada uso de chave é registrado - sabemos quem acessou, quando e por quê |
| 🔄 **Rotação** | Chaves são trocadas periodicamente automaticamente |

**Exemplo prático:** Imagine que o cofre (AES-256) protege seus dados, mas a chave do cofre também precisa ser protegida. O AWS KMS é "o cofre da chave do cofre" - com segurança de nível militar.

### Por que isso importa?

- **Para candidatos:** Suas informações estão protegidas tanto quando armazenadas quanto quando transmitidas pela internet
- **Para clientes:** Podem garantir a fornecedores e reguladores que usam uma plataforma com criptografia de ponta

### 4.4.2 📱 MFA (Autenticação Multi-Fator)

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Exigir mais de uma "prova" de identidade para fazer login - algo que você sabe (senha) + algo que você tem (celular) |
| 🎯 **Para que serve?** | Impedir acesso mesmo se alguém descobrir sua senha |
| 🔢 **Fatores comuns** | Senha + código no celular, biometria, chave de segurança física |
| 👥 **Quem precisa?** | Obrigatório para administradores, recomendado para todos os usuários |
| 🛡️ **O que protege?** | Ataques de phishing, vazamento de senhas, acessos não autorizados |

**Os 3 Fatores de Autenticação:**
| 🔑 Fator | 📋 Exemplo | 🛡️ Proteção |
|---------|----------|------------|
| 📝 **Algo que você sabe** | Senha, PIN | Básico - pode ser roubado |
| 📱 **Algo que você tem** | Celular, token, chave física | Forte - atacante precisa do dispositivo |
| 👤 **Algo que você é** | Impressão digital, face | Muito forte - exclusivo de você |

**Exemplo prático:** Um hacker descobre a senha de um administrador através de phishing. Ao tentar fazer login, o sistema pede o código do aplicativo autenticador. O hacker não tem o celular do administrador → acesso negado.

**Métodos de MFA suportados:**
| 📱 Método | 🛡️ Segurança | 📋 Descrição |
|----------|------------|-------------|
| 📲 App Autenticador (TOTP) | ⭐⭐⭐⭐ | Google Authenticator, Authy - códigos que mudam a cada 30s |
| 📨 SMS | ⭐⭐ | Código por mensagem de texto (menos seguro, mas melhor que nada) |
| 🔑 Chave de Segurança | ⭐⭐⭐⭐⭐ | YubiKey, Titan - dispositivo físico (mais seguro) |

### 4.4.3 👥 RBAC (Role-Based Access Control)

| 📋 **Exemplo de papéis** | Administrador, Recrutador, Hiring Manager, Visualizador |
| 🔄 **Flexibilidade** | Papéis podem ser combinados; permissões podem ser customizadas |
| 📊 **Auditoria** | Fácil verificar quem tem acesso a quê |

**Papéis na WeDo Talent:**

**Benefícios do RBAC:**

| 🔄 **Escalabilidade** | Fácil dar permissões a novos funcionários (só atribuir papel) |

## 5.4 📊 Modelo de Tratamento de Dados

> ⚠️ **IMPORTANTE: Quem é o dono dos dados?**
>
> A WeDo Talent **NÃO é proprietária dos dados de candidatos**. Atuamos como **OPERADORA** (processadora) de dados em nome dos nossos clientes (empresas contratantes), que são os **CONTROLADORES** dos dados conforme a LGPD.

### 5.4.1 🏛️ Papéis na LGPD

| 👤 Papel | 🏢 Quem é? | 📋 Responsabilidades |
|---------|----------|---------------------|
| 🎯 **CONTROLADOR** | **Empresa cliente** (quem contrata a WeDo Talent) | Define finalidades do tratamento, políticas de retenção, bases legais, responde aos titulares |
| ⚙️ **OPERADOR** | **WeDo Talent** | Processa dados seguindo instruções do controlador, implementa medidas de segurança, não decide sobre uso dos dados |
| 👤 **TITULAR** | **Candidato** | Pessoa física cujos dados são tratados no processo seletivo |

### 5.4.2 📋 O que isso significa na prática?

| ❓ Pergunta | 📝 Resposta |
|------------|-------------|
| 🏢 **De quem são os dados dos candidatos?** | São do **cliente (empresa contratante)**, não da WeDo Talent |
| 📥 **Quem coleta os dados?** | O **cliente** coleta, seja via formulário próprio, LinkedIn, ou outras fontes. A WeDo Talent apenas processa |
| ⏱️ **Quem define o tempo de retenção?** | O **cliente** define suas políticas de retenção. A WeDo Talent executa conforme instruções |
| 🗑️ **Quem decide sobre exclusão?** | O **cliente** autoriza exclusões. Candidatos devem solicitar ao cliente (empresa) |
| 📤 **A WeDo Talent compartilha dados?** | **NÃO**. Os dados já são do cliente. A WeDo Talent apenas processa em nome dele |
| 🔐 **Quem define as bases legais?** | O **cliente** define (consentimento, legítimo interesse, etc.) conforme sua política de privacidade |

> 💡 **O que são os Princípios da LIA?**
>
> Os Cinco Princípios da LIA são os valores fundamentais que guiam todo o desenvolvimento e uso da nossa Inteligência Artificial. Eles foram criados para garantir que a tecnologia trabalhe A FAVOR de candidatos e empresas, nunca contra. Cada princípio tem impacto direto na forma como você é avaliado e como as empresas conduzem seus processos seletivos.

### Resumo dos Princípios

| # | 🌟 Princípio | 📋 Resumo |
|---|-------------|----------|
| 1 | 🌈 **Diversidade** | Equipe diversa constrói IA mais justa |
| 2 | 🚫 **Anti-Discriminação** | Campos protegidos nunca são usados |
| 3 | 🔍 **Transparência** | Todas as decisões são explicáveis |
| 4 | 🔐 **Privacidade & Segurança** | Apenas dados profissionais são avaliados |
| 5 | ⚖️ **Responsabilidade** | Comitê de Ética supervisiona a IA |

---

### 7.1.1 🌈 Diversidade

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Nosso compromisso de ter uma equipe de desenvolvimento composta por pessoas de diferentes gêneros, raças, origens, idades e experiências |
| 🎯 **Para que serve?** | Equipes diversas identificam vieses que equipes homogêneas não percebem - diferentes perspectivas criam IA mais justa |
| 🔬 **Base científica** | Pesquisas mostram que equipes diversas produzem software com menos viés algorítmico |
| 👥 **Na prática** | Pessoas de diferentes backgrounds revisam prompts, testam algoritmos e validam resultados |
| 🛡️ **O que protege?** | Evita que a IA reproduza preconceitos inconscientes de um único grupo demográfico |

> 👥 Nossa equipe de desenvolvimento é composta por pessoas de diferentes gêneros, raças, origens e experiências. Essa pluralidade ajuda a construir uma IA mais justa e representativa.

**Exemplo prático:** Quando criamos perguntas para avaliar "liderança", pessoas de diferentes culturas revisam para garantir que a definição não seja enviesada para um único estilo de liderança (ex: apenas o estilo "ocidental" ou "masculino").

**Riscos de discriminação minimizados:**
| ❌ Risco | ✅ Como a diversidade mitiga |
|---------|---------------------------|
| IA favorece candidatos similares aos desenvolvedores | Equipe diversa identifica e corrige vieses de "afinidade" |
| Linguagem das avaliações exclui grupos | Pessoas de diferentes backgrounds revisam a comunicação |
| Critérios culturalmente tendenciosos | Múltiplas perspectivas validam se os critérios são universais |

**Por que isso importa?**
- **Para candidatos:** Você é avaliado por uma IA construída por pessoas que entendem diferentes realidades - não apenas por um grupo homogêneo
- **Para clientes:** Demonstra compromisso genuíno com D&I (Diversidade e Inclusão) desde a tecnologia que usam

### 7.1.2 🚫 Anti-Discriminação

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | O compromisso de que a LIA NUNCA utilizará características pessoais protegidas para avaliar candidatos |
| ⚖️ **Base legal** | LGPD Art. 20, Lei 9.029/1995, CF Art. 5º, EU AI Act, NYC LL144 |
| 🎯 **Para que serve?** | Garantir que a avaliação seja baseada APENAS em competências profissionais relevantes para a vaga |
| 🔧 **Como funciona?** | Campos protegidos são excluídos do processamento antes mesmo de chegar à IA |
| 🛡️ **O que protege?** | Impede discriminação direta e indireta por características pessoais |

> ⛔ LIA é configurada para evitar discriminação. Campos protegidos são excluídos de qualquer análise:

| ❌ Campos EXCLUÍDOS da Análise | 📋 Por que é excluído? | ⚖️ Base Legal |
|------------------------------|----------------------|--------------|
| 👤 Gênero | Não influencia capacidade profissional | Lei 9.029/1995 |
| 🎂 Idade | Experiência ≠ idade cronológica | Lei 9.029/1995 |
| 🌍 Etnia/Raça | Irrelevante para competências | CF Art. 5º |
| 💍 Estado civil | Vida pessoal não afeta trabalho | Lei 9.029/1995 |
| 🏳️‍🌈 Orientação sexual | Privacidade e dignidade | CF Art. 5º |
| ⛪ Religião | Liberdade religiosa protegida | CF Art. 5º |
| ♿ Deficiência | Foco em capacidades, não limitações | Lei 13.146/2015 |
| 🤰 Gravidez | Proteção à maternidade | CLT Art. 373-A |

**Como funciona tecnicamente:**

| 🔄 Etapa | 📋 O que acontece |
|---------|------------------|
| 1️⃣ **Ingestão** | Currículo/dados do candidato chegam à plataforma |
| 2️⃣ **Sanitização** | Campos protegidos são identificados e REMOVIDOS antes do processamento |
| 3️⃣ **Processamento** | IA analisa APENAS dados profissionais (experiência, formação, habilidades) |
| 4️⃣ **Saída** | Avaliação gerada sem qualquer referência a características protegidas |

**Exemplo prático:** Se um currículo menciona "Presidente do grupo de mulheres na empresa X", a IA analisa apenas a habilidade de liderança e organização, não o gênero implícito na atividade.

**Tipos de discriminação que evitamos:**

| 🚫 Tipo | 📋 Descrição | 🛡️ Como evitamos |
|--------|-------------|-----------------|
| **Direta** | Rejeitar por ser mulher/negro/idoso | Campos protegidos excluídos |
| **Indireta** | Critérios "neutros" que prejudicam grupos | Auditoria de impacto disparatado |
| **Por associação** | Prejudicar por estar associado a grupo | Análise apenas de competências individuais |

**Por que isso importa?**
- **Para candidatos:** Certeza de que sua avaliação depende APENAS de suas competências, não de quem você é
- **Para clientes:** Proteção jurídica contra ações trabalhistas por discriminação, demonstração de compliance

### 7.1.3 🔍 Transparência

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | O compromisso de que todas as decisões da IA são explicáveis em linguagem compreensível |
| ⚖️ **Base legal** | LGPD Art. 20 - direito à explicação de decisões automatizadas |
| 🎯 **Para que serve?** | Permitir que candidatos entendam POR QUE receberam determinada avaliação |
| 👤 **Quem pode solicitar?** | Qualquer candidato avaliado pela LIA |
| ⏱️ **Prazo de resposta** | Até 15 dias corridos (conforme LGPD) |

> 📋 Todas as decisões da IA são explicáveis. Candidatos podem solicitar explicação detalhada de sua avaliação a qualquer momento, conforme LGPD Art. 20.

**O que a explicação inclui:**

| 📊 Componente | 📋 O que você recebe |
|--------------|---------------------|
| 🎯 **Critérios avaliados** | Quais competências foram analisadas |
| 📈 **Pontuação por critério** | Como você foi avaliado em cada aspecto |
| ✅ **Pontos fortes** | O que a IA identificou como qualidades |
| ⚠️ **Gaps identificados** | Onde há oportunidades de desenvolvimento |
| 🔧 **Metodologia** | Qual método foi usado (WSI, Rubric, etc.) |
| 📚 **Dados utilizados** | Quais informações do seu perfil foram consideradas |

**Exemplo de explicação (simplificado):**

```
📊 Avaliação para: Desenvolvedor Full Stack Sênior

🎯 Critérios Avaliados:
- Experiência com React: 4.5/5.0 (8 anos de experiência documentada)
- Backend Node.js: 4.0/5.0 (múltiplos projetos relevantes)
- Liderança técnica: 3.8/5.0 (experiência como tech lead por 2 anos)

✅ Pontos Fortes:
- Sólida experiência em frontend moderno
- Certificações atualizadas
- Histórico de projetos escaláveis

⚠️ Gaps:
- Experiência limitada com Kubernetes (requisito desejável)

📊 Score Final: 85/100 - Recomendação: Entrevista Técnica
```

**Por que isso importa?**
- **Para candidatos:** Você entende a lógica por trás da avaliação e pode melhorar para futuras oportunidades
- **Para clientes:** Demonstra que o processo é objetivo e justificável, facilitando feedbacks para candidatos

### 7.1.4 🔐 Privacidade & Segurança

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | O compromisso de que dados pessoais sensíveis NUNCA são usados para avaliação de candidatos |
| ⚖️ **Base legal** | LGPD Art. 11 - tratamento de dados sensíveis |
| 🎯 **Para que serve?** | Proteger a privacidade do candidato e garantir avaliação justa baseada em competências |
| 🛡️ **O que protege?** | Impede que informações pessoais influenciem decisões de recrutamento |

> 🛡️ Dados pessoais sensíveis nunca são usados para avaliação. Usamos apenas:

**Dados que a IA UTILIZA:**

| ✅ Dado Utilizado | 📋 Por que é relevante | 🎯 Como afeta a avaliação |
|-----------------|----------------------|-------------------------|
| 💼 **Experiência profissional** | Demonstra competências práticas | Peso na aderência à vaga |
| 🎓 **Formação acadêmica** | Conhecimento teórico fundamentante | Avaliação de base técnica |
| 🔧 **Habilidades técnicas** | Capacidade de executar o trabalho | Match com requisitos |
| 📜 **Certificações** | Validação externa de competências | Pontos adicionais |
| 🌍 **Idiomas** | Comunicação necessária para a função | Requisito funcional |
| 🏆 **Conquistas profissionais** | Evidência de resultados | Diferenciação positiva |

**Dados que a IA NÃO UTILIZA:**

| ❌ Dado Excluído | 📋 Por que não é usado |
|----------------|----------------------|
| 📷 Foto | Pode revelar características protegidas |
| 🏠 Endereço completo | Pode indicar classe social/região |
| 💰 Pretensão salarial | Negociação é etapa posterior |
| 🔗 Redes sociais pessoais | Vida pessoal ≠ capacidade profissional |
| 📊 Score de crédito | Irrelevante para competências |

**Por que isso importa?**
- **Para candidatos:** Seus dados pessoais são protegidos e nunca usados contra você
- **Para clientes:** Garantia de que estão usando dados apenas para fins legítimos de recrutamento

## 7.2 🔬 Metodologias de Avaliação

> 💡 **O que são as Metodologias de Avaliação?**
>
> As metodologias de avaliação são os MÉTODOS CIENTÍFICOS que a LIA utiliza para analisar candidatos de forma justa, consistente e auditável. Cada metodologia foi desenvolvida com base em pesquisas acadêmicas validadas e serve a um propósito específico no processo seletivo. Aqui explicamos cada uma em detalhes para que você entenda exatamente como é avaliado.

### Resumo das Metodologias

| # | 🔬 Metodologia | 📋 O que avalia | 🎯 Quando é usada |
|---|---------------|----------------|------------------|
| 1 | 📊 **WSI** | Competências via conversa | Triagem conversacional |
| 2 | 📑 **Rubric** | CV vs requisitos da vaga | Análise de currículo |
| 3 | 🎯 **LIA Scoring** | Score numérico consolidado | Ranking de candidatos |
| 4 | 📝 **LIA Opinion** | Parecer qualitativo | Feedback para recrutador |

---

### 7.2.1 📊 WSI (WeDoTalent Skill Index)

| 📌 Aspecto | 📋 Descrição |
|-----------|-------------|
| ❓ **O que é?** | Um índice que mede competências técnicas e comportamentais através de uma conversa estruturada com a LIA |
| 🎯 **Para que serve?** | Avaliar candidatos de forma padronizada, justa e baseada em evidências comportamentais |
| 📚 **Base Teórica** | Entrevista por Competências (McClelland), Taxonomia de Bloom, Modelo Dreyfus, Big Five |
| ⏱️ **Duração** | 5-10 minutos de conversa |
| ⚖️ **Distribuição** | 70% competências técnicas, 30% comportamentais |

> 🔬 O WSI combina os melhores frameworks de avaliação de competências em um único índice conversacional.

**Fundamentos científicos do WSI:**

| 📚 Teoria | 👤 Autor | 📋 Contribuição para o WSI |
|----------|---------|--------------------------|
| 🎯 **CBI** | David McClelland | Avaliação baseada em comportamentos passados como preditores de performance futura |
| 📊 **Taxonomia de Bloom** | Benjamin Bloom | Níveis de profundidade do conhecimento (lembrar → criar) |
| 🔄 **Modelo Dreyfus** | Stuart & Hubert Dreyfus | Estágios de proficiência (novato → expert) |
| 🧠 **Big Five** | Costa & McCrae | Traços de personalidade relevantes para o trabalho |

**Como funciona na prática:**

| 🔄 Etapa | 📋 O que acontece | 🎯 O que é avaliado |
|---------|------------------|-------------------|
| 1️⃣ | LIA faz pergunta situacional | Abertura para demonstrar competência |
| 2️⃣ | Candidato responde com exemplo real | Evidência comportamental (STAR) |
| 3️⃣ | LIA faz perguntas de aprofundamento | Profundidade do conhecimento |
| 4️⃣ | Respostas são analisadas por IA | Extração de indicadores objetivos |
| 5️⃣ | Score é calculado por competência | Média ponderada final |

**Exemplo de pergunta WSI:**
```
💬 LIA: "Conte-me sobre uma situação em que você precisou resolver 
um problema técnico complexo sob pressão de prazo."

📋 O que a IA avalia na resposta:
- Situação: Contexto claro e específico
- Tarefa: Responsabilidade individual
- Ação: Passos técnicos tomados
- Resultado: Impacto mensurável
```

**🚦 Faixas de Decisão:**

| 📊 Score | 🏷️ Resultado | 📋 O que significa |
|---------|-------------|-------------------|
| ≥ 4.2 | ✅ **Aprovado automático** | Candidato demonstrou competências excepcionais - avança no processo |
| 3.8-4.1 | 👤 **Revisão humana** | Resultado limítrofe - recrutador decide com base no contexto |
| < 3.8 | ❌ **Não recomendado** | Evidências insuficientes - candidato recebe feedback construtivo |

**Por que o WSI é justo:**

| ✅ Característica | 📋 Como beneficia candidatos |
|-----------------|---------------------------|
| 🎯 **Padronizado** | Todos os candidatos respondem às mesmas perguntas |
| 📊 **Objetivo** | Avaliação baseada em evidências, não impressões |
| 🔄 **Consistente** | Mesmos critérios aplicados independente do avaliador |
| 📋 **Transparente** | Candidato recebe explicação do resultado |

**Por que isso importa?**
- **Para candidatos:** Avaliação justa baseada no que você DEMONSTROU saber fazer, não em impressões subjetivas
- **Para clientes:** Metodologia validada cientificamente que reduz risco de contratações equivocadas

### 7.2.2 📑 Rubric Evaluation (Avaliação por Rubrica - CV vs Vaga)

| 📌 Aspecto | 📋 Descrição |
|-----------|-------------|
| ❓ **O que é?** | Uma análise estruturada que compara o currículo do candidato com os requisitos específicos da vaga |
| 🎯 **Para que serve?** | Identificar objetivamente o quanto um candidato atende aos requisitos listados |
| 🔧 **Método** | Cada requisito é avaliado individualmente numa escala de 1-5 |
| 📋 **Critérios** | Experiência, formação, habilidades técnicas, certificações, idiomas |

> 📑 A Rubric Evaluation elimina a subjetividade da análise de currículos, criando um processo justo e auditável.

**Como funciona:**

| 🔄 Etapa | 📋 O que acontece |
|---------|------------------|
| 1️⃣ **Extração** | LIA extrai informações estruturadas do currículo |
| 2️⃣ **Mapeamento** | Requisitos da vaga são listados como critérios |
| 3️⃣ **Comparação** | Cada critério é comparado com o perfil do candidato |
| 4️⃣ **Pontuação** | Nota de 1-5 atribuída por critério com justificativa |
| 5️⃣ **Consolidação** | Média ponderada gera score final |

**Escala de Pontuação:**

| 📊 Nota | 🏷️ Significado | 📋 Critério |
|--------|---------------|-----------|
| ⭐ **1** | Não atende | Nenhuma evidência do requisito |
| ⭐⭐ **2** | Atende parcialmente | Experiência básica ou indireta |
| ⭐⭐⭐ **3** | Atende | Experiência adequada para o requisito |
| ⭐⭐⭐⭐ **4** | Supera | Experiência acima do esperado |
| ⭐⭐⭐⭐⭐ **5** | Excepcional | Referência na área |

**Exemplo de Rubric Evaluation:**

| 📋 Requisito da Vaga | 📊 CV do Candidato | 🏷️ Nota | 📝 Justificativa |
|---------------------|-------------------|--------|-----------------|
| React 3+ anos | 5 anos com React | ⭐⭐⭐⭐ 4 | Supera requisito mínimo |
| TypeScript | Mencionado em 2 projetos | ⭐⭐⭐ 3 | Atende |
| Inglês fluente | Intermediário | ⭐⭐ 2 | Atende parcialmente |
| Certificação AWS | Não mencionada | ⭐ 1 | Não atende |

**Diferença entre Rubric e análise manual:**

| ❌ Análise Manual | ✅ Rubric Evaluation |
|------------------|---------------------|
| Varia conforme quem analisa | Critérios padronizados |
| Pode favorecer currículos "bonitos" | Foca apenas no conteúdo |
| Difícil de auditar | Totalmente rastreável |
| Impressões subjetivas | Pontuação justificada |

**Por que isso importa?**
- **Para candidatos:** Seu currículo é avaliado pelos mesmos critérios que todos os outros candidatos
- **Para clientes:** Análise objetiva e justificável que pode ser auditada e defendida

### 7.2.3 🎯 LIA Scoring

| 📌 Aspecto | 📋 Descrição |
|-----------|-------------|
| ❓ **O que é?** | Um score numérico consolidado (0-100) que representa a aderência geral do candidato à vaga |
| 🎯 **Para que serve?** | Criar um ranking objetivo de candidatos para facilitar a triagem |
| 📊 **Escala** | 0-100 pontos |
| 🔧 **Fatores** | Aderência técnica, experiência, formação, fit com a empresa |

> 🎯 O LIA Score é a consolidação de todas as avaliações em um único número comparável.

**Composição do LIA Score:**

| 📊 Fator | ⚖️ Peso | 📋 O que avalia |
|---------|--------|----------------|
| 🔧 **Aderência Técnica** | 40% | Match entre habilidades e requisitos técnicos |
| 💼 **Experiência** | 30% | Relevância e profundidade da experiência profissional |
| 🎓 **Formação** | 15% | Educação formal e certificações |
| 🏢 **Fit Cultural** | 15% | Alinhamento com valores e cultura da empresa |

**Faixas de Score:**

| 📊 Faixa | 🏷️ Classificação | 📋 Recomendação |
|---------|-----------------|----------------|
| 90-100 | 🌟 **Excepcional** | Prioridade máxima - entrevistar imediatamente |
| 75-89 | ✅ **Alto potencial** | Forte candidato - agendar entrevista |
| 60-74 | 🟡 **Potencial** | Avaliar junto com outros fatores |
| 40-59 | ⚠️ **Parcial** | Gaps significativos - considerar para outras vagas |
| 0-39 | ❌ **Baixa aderência** | Perfil não aderente à vaga atual |

**Como o score é calculado (exemplo):**

```
📊 Candidato: Ana Silva para Desenvolvedora Full Stack Senior

Aderência Técnica (40%):
  - React: 95/100 × 0.40 = 38 pontos
  
Experiência (30%):  
  - 8 anos relevantes: 85/100 × 0.30 = 25.5 pontos
  
Formação (15%):
  - Pós-graduação + certificações: 80/100 × 0.15 = 12 pontos
  
Fit Cultural (15%):
  - Empresa prévia similar: 75/100 × 0.15 = 11.25 pontos

📈 LIA Score Final: 86.75 → 87/100 ✅ Alto Potencial
```

**Por que o LIA Score é justo:**

| ✅ Característica | 📋 Benefício |
|-----------------|-------------|
| 📊 **Objetivo** | Mesmo cálculo para todos os candidatos |
| ⚖️ **Ponderado** | Pesos refletem importância real para a vaga |
| 🔍 **Transparente** | Candidato pode ver a composição do score |
| 🔄 **Ajustável** | Pesos podem ser calibrados por vaga |

**Por que isso importa?**
- **Para candidatos:** Você sabe exatamente como foi calculada sua pontuação e o que pode melhorar
- **Para clientes:** Facilita a triagem objetiva de grandes volumes de candidatos

### 7.2.4 📝 LIA Opinion (Parecer Qualitativo)

| 📌 Aspecto | 📋 Descrição |
|-----------|-------------|
| ❓ **O que é?** | Um parecer em texto, escrito em linguagem natural, que resume a avaliação do candidato |
| 🎯 **Para que serve?** | Fornecer contexto qualitativo além dos números para apoiar a decisão do recrutador |
| 📝 **Formato** | Texto estruturado com pontos fortes, gaps e recomendação |
| 👤 **Destinatário** | Recrutadores e hiring managers |

> 📝 O LIA Opinion traduz dados em insights acionáveis para tomada de decisão humana.

**Estrutura do Parecer:**

| 📋 Seção | 📝 Conteúdo |
|---------|-----------|
| 📊 **Resumo Executivo** | Visão geral em 2-3 frases |
| ✅ **Pontos Fortes** | 3-5 qualidades destacadas com evidências |
| ⚠️ **Gaps/Riscos** | Áreas de atenção ou desenvolvimento necessário |
| 💡 **Recomendação** | Próximo passo sugerido (entrevistar, avaliar mais, etc.) |
| ❓ **Perguntas Sugeridas** | Tópicos para aprofundar na entrevista |

**Exemplo de LIA Opinion:**

```
📝 PARECER LIA - Ana Silva para Desenvolvedora Full Stack Senior

📊 RESUMO EXECUTIVO:
Profissional sênior com 8 anos de experiência sólida em React e Node.js. 
Histórico de liderança técnica e projetos de alta escalabilidade. 
Forte aderência técnica com gap em experiência Kubernetes.

✅ PONTOS FORTES:
• Experiência excepcional em React (5 anos, incluindo projetos enterprise)
• Histórico comprovado de liderança técnica (2 anos como Tech Lead)
• Certificações atualizadas em AWS e React
• Comunicação clara evidenciada nas respostas

⚠️ GAPS/RISCOS:
• Experiência limitada com Kubernetes (requisito desejável)
• Não menciona experiência com GraphQL (requisito secundário)

💡 RECOMENDAÇÃO: ✅ ENTREVISTAR
Candidata com forte aderência. Gaps são em requisitos desejáveis, 
não eliminatórios. Recomendo entrevista técnica focada em 
arquitetura de microsserviços.
```

## 7.3 👤 Supervisão Humana

> 💡 **O que é Supervisão Humana?**
>
> Supervisão humana significa que a IA da WeDo Talent NUNCA toma decisões finais sobre candidatos. Toda recomendação da LIA é revisada por um ser humano antes de qualquer ação. Isso garante que a tecnologia seja uma FERRAMENTA de apoio, não um juiz autônomo.

### Por que a IA não decide sozinha?

| 📌 Motivo | 📋 Explicação |
|----------|--------------|
| ⚖️ **Legal** | EU AI Act Art. 14 exige supervisão humana para IA de alto risco |
| 🧠 **Contextual** | Humanos captam nuances que algoritmos podem perder |
| 🛡️ **Ético** | Decisões que afetam vidas merecem julgamento humano |
| 🔄 **Correção** | Humanos podem corrigir erros da IA em tempo real |
| 🤝 **Confiança** | Candidatos confiam mais em processos com humanos no loop |

> 🛡️ Conforme EU AI Act Art. 14, garantimos controle humano em todas as decisões:

**Garantias de Supervisão Humana:**

| ✅ Garantia | 📋 Descrição | 🛡️ O que protege |
|-----------|-------------|-----------------|
| 🚫 **Sem decisão final** | IA nunca decide contratação ou rejeição | Candidato sempre tem chance com humano |
| 👤 **Revisão obrigatória** | Recrutador revisa TODAS as recomendações | Nenhuma decisão passa despercebida |
| 🆘 **Revisão sob demanda** | Candidatos podem pedir revisão humana | Direito garantido a qualquer momento |
| ⚡ **Override disponível** | Humano pode sobrescrever qualquer decisão da IA | Última palavra é sempre humana |

**Fluxo de Supervisão Humana:**

| 🔄 Etapa | 🤖 IA | 👤 Humano |
|---------|------|---------|
| 1️⃣ Triagem inicial | Analisa e recomenda | Revisa recomendações |
| 2️⃣ Scoring | Calcula pontuação | Valida ou ajusta |
| 3️⃣ Parecer | Gera LIA Opinion | Lê e contextualiza |
| 4️⃣ Decisão | Sugere próximo passo | DECIDE ação final |
| 5️⃣ Comunicação | Prepara rascunho | Aprova e envia |

**Cenários de Override (quando humano sobrescreve IA):**

| 🔄 Situação | 🤖 IA recomendou | 👤 Humano decidiu | 📋 Motivo |
|------------|-----------------|------------------|----------|
| Experiência não-tradicional | ❌ Reprovar | ✅ Entrevistar | Candidato tem skills transferíveis |
| Gap recente explicável | ⚠️ Risco | ✅ Aprovar | Sabbatical justificado |
| Fit cultural duvidoso | 🟡 Revisar | ✅ Aprovar | Diversidade de pensamento valorizada |
| Overqualificação | ⚠️ Risco de turnover | ✅ Aprovar | Candidato busca nova fase de carreira |

**Direito à Revisão Humana:**

| 📌 Aspecto | 📋 Detalhes |
|-----------|-------------|
| 👤 **Quem pode solicitar?** | Qualquer candidato avaliado pela LIA |
| ⏱️ **Quando solicitar?** | A qualquer momento durante o processo |
| 📍 **Como solicitar?** | Portal do candidato ou email para suporte |
| ⏱️ **Prazo de resposta** | Até 5 dias úteis |
| 📋 **O que acontece?** | Recrutador humano reavalia todo o processo |

**Por que isso importa?**
- **Para candidatos:** Você NUNCA será rejeitado apenas por uma máquina - sempre há um humano responsável
- **Para clientes:** Proteção jurídica e ética, processo mais robusto e defensável

## 7.4 🎯 Mitigação de Viés

> 💡 **O que é Mitigação de Viés?**
>
> Mitigação de viés são as técnicas e controles que usamos para PREVENIR que a IA reproduza ou amplifique preconceitos humanos. Sistemas de IA aprendem com dados históricos que podem conter vieses - nosso trabalho é identificar e neutralizar esses vieses antes que afetem candidatos.

### Por que viés em IA é um risco?

| ⚠️ Problema | 📋 Exemplo | 🛡️ Consequência se não mitigado |
|------------|----------|-------------------------------|
| 📊 **Viés histórico** | Dados de contratações passadas favorecem certo perfil | IA perpetua discriminação histórica |
| 🔧 **Viés de proxy** | CEP usado como proxy para raça/classe social | Discriminação indireta |
| 📝 **Viés de linguagem** | Termos "agressivo" vs "assertivo" por gênero | Avaliação diferente para mesma competência |
| 🏢 **Viés de afinidade** | Favorece candidatos de empresas "de prestígio" | Exclui talentos de backgrounds diversos |

**Controles Implementados:**

| 🛡️ Controle | ⚙️ Implementação | 📋 O que faz |
|------------|-----------------|-------------|
| ⛔ **Campos Protegidos** | Exclusão de gênero, idade, etnia, orientação, religião, deficiência | Impede discriminação direta |
| 🤖 **Prompts Anti-Viés** | Instruções explícitas nos prompts de IA | Direciona modelo para justiça |
| 📊 **Auditoria Anual** | Planejada com Warden AI (Q3 2026) | Validação externa independente |
| 📈 **Monitoramento** | Acompanhamento de disparate impact | Detecta viés emergente |

---

### 7.4.1 ⛔ Exclusão de Campos Protegidos

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | A remoção sistemática de características protegidas por lei antes de qualquer processamento de IA |
| 🎯 **Para que serve?** | Garantir que a IA não tenha acesso a informações que possam causar discriminação |
| 🔧 **Como funciona?** | Algoritmos de sanitização removem ou mascaram campos sensíveis |
| ⚖️ **Base legal** | LGPD, Lei 9.029/1995, CF Art. 5º e 7º |

**Campos excluídos e técnicas de remoção:**

| ❌ Campo | 🔧 Técnica de Remoção | 🛡️ O que evita |
|---------|----------------------|---------------|
| 👤 Gênero | Remoção de pronomes e nomes | Viés de gênero |
| 🎂 Idade | Exclusão de datas de nascimento e formatura | Etarismo |
| 🌍 Etnia/Raça | Remoção de indicadores culturais | Racismo |
| 📍 Localização detalhada | Uso apenas de região/cidade | Classismo geográfico |
| 📷 Foto | Nunca processada pela IA | Todos os vieses visuais |

---

### 7.4.2 🤖 Prompts Anti-Viés

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Instruções explícitas inseridas nos prompts da IA para direcionar comportamento justo |
| 🎯 **Para que serve?** | "Programar" a IA para evitar discriminação mesmo em situações ambíguas |
| 🔧 **Como funciona?** | Cada prompt contém diretrizes de fairness que a IA deve seguir |

**Exemplos de instruções anti-viés:**

```
📝 TRECHO DE PROMPT REAL (simplificado):

"Ao avaliar candidatos:
- NÃO considere nomes como indicadores de gênero, etnia ou origem
- NÃO infira idade a partir de anos de formatura ou experiência
- NÃO associe gaps de carreira a falta de comprometimento
- NÃO favoreça candidatos de universidades ou empresas específicas
- AVALIE apenas competências relevantes para a função
- DOCUMENTE o racional de cada pontuação
- APLIQUE os mesmos critérios para todos os candidatos"
```

### 7.4.4 📈 Monitoramento Contínuo (Disparate Impact)

| 📌 Aspecto | 📋 Explicação |
|-----------|--------------|
| ❓ **O que é?** | Acompanhamento estatístico contínuo para detectar se a IA está gerando resultados desiguais entre grupos |
| 🎯 **Para que serve?** | Identificar viés ANTES que cause danos significativos |
| 📊 **Métrica principal** | Regra dos 80% (Four-Fifths Rule) |
| 🚨 **Alertas** | Sistema notifica equipe quando detecta anomalias |

**Regra dos 80% (Four-Fifths Rule):**

> A taxa de seleção do grupo minoritário deve ser pelo menos 80% da taxa do grupo majoritário.

| 📊 Exemplo | 📋 Cálculo | 🚦 Resultado |
|-----------|-----------|-------------|
| Homens selecionados: 50% | Base de comparação | - |
| Mulheres selecionadas: 45% | 45% ÷ 50% = 90% | ✅ OK (≥80%) |
| Mulheres selecionadas: 35% | 35% ÷ 50% = 70% | ⚠️ Alerta (< 80%) |

**Por que isso importa?**
- **Para candidatos:** Monitoramento ativo garante que você não seja prejudicado por vieses sistêmicos
- **Para clientes:** Proteção proativa contra riscos legais e reputacionais de discriminação algorítmica
