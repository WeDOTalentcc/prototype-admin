# LIA Observability Markers — Reference

> Human-readable summary of all `[LIA-*]` log markers emitted by the LIA runtime.
> **Source of truth:** `app/shared/observability/marker_catalog.yaml` — this doc
> is derived. Edits to marker semantics go to the YAML; this doc is rebuilt.
>
> **Maintained by:** Track 3 G2 CI guard (`tests/unit/test_g2_marker_catalog.py`).
> If you add a new `[LIA-XXX]` log line in code, CI will fail until you add an
> entry to `marker_catalog.yaml`.

## How to use

### On-call / debugging

When investigating a chat issue, grep uvicorn logs for relevant marker:

```bash
# Memory setup failures
grep '\[LIA-M01\]' /tmp/uvicorn.log

# Compliance fail-open (silent degradation)
grep -E '\[LIA-(C06|P01|P02|P03|P04)\]' /tmp/uvicorn.log

# Agentic tool calling decisions (most active path)
grep '\[LIA-A04\]' /tmp/uvicorn.log
```

### Dashboard authors

Categories map naturally to Grafana panels:

- **Agent health:** A01, A02, A04 (tool-calling + phase decisions)
- **Compliance drift:** C01, C02, C06, C07 (architectural + LGPD fails)
- **Routing accuracy:** I04, I05 (intent/domain disagreements — eval signal)
- **Memory subsystem:** M01, M02, M04, M05 (M05 is P0 if fires in prod)
- **Pipeline fail-open:** P01-P04 (silent skips that could mask bugs)

## Severity conventions

- `error` — real failures that users feel (e.g., M05 in-memory state in prod)
- `warning` — degraded behavior, deserves alert (C01, C07, M01, M02)
- `info` — useful signal for analysis (I02, I03, E04, BYOK)
- `debug` — noisy / fail-open guards (A01, A02, P01-P04)

## P0 markers (page on-call if firing)

- **LIA-C07** company_id mismatch (multi-tenancy breach attempt)
- **LIA-M05** StateManager in-memory fallback in production (Redis down)

## FIX-level log prefixes (not `[LIA-*]`)

FIX 20-32 emit informative log lines tagged with their FIX number:

- `FIX 20 search_jobs DB result` — pagination visible per query
- `FIX 22 close_job reason normalized` — enum PT→EN mapping fired
- `FIX 26/30 quantifier` — memory_resolver enriched short replies
- `FIX 31 memory enrichment applied` — resolver ran in production chat path

See `marker_catalog.yaml` `fix_log_prefixes` section for full list.

## Adding a new marker

1. Add the logger line in code: `logger.info("[LIA-NEW] ...", ...)`
2. Add entry to `app/shared/observability/marker_catalog.yaml`:
   ```yaml
   LIA-NEW:
     category: <agent|compliance|memory|routing|pipeline|quality>
     purpose: "One-line description"
     origin: "app/path/to/emitter.py"
     severity: <debug|info|warning|error>
   ```
3. Run `pytest tests/unit/test_g2_marker_catalog.py` — should pass.
4. Commit both code + YAML together. CI drift guard prevents orphan markers.
