Exemplo Real: Fluxo de Triagem de Candidato — Passo a Passo
Cenário: Um recrutador pede ao agente: "Avalie o candidato João Silva para a vaga de Engenheiro Senior". O que acontece em cada arquitetura? Quais componentes são ativados? Quais proteções são aplicadas?
v5 — Fluxo de Triagem (evaluation domain)
1
HTTP Request chega ao backend

src/api/routes.py
↓
2
Orchestrator Hub recebe e roteia
orchestrator.py (626L) classifica o intent e identifica: domínio = evaluation
src/orchestrator/orchestrator.py → SupervisorGraph
IntentClassifier DomainCatalog ContextBuilder FairnessGuard PII Masking
↓
3
DomainWorkflow Pipeline (3 nos LangGraph)
classify_intent → execute_action → format_response. Sem fairness check neste pipeline.
src/orchestrator/domain_workflow.py
LangGraph 3 nos GuardrailCheck AuditTrail
↓
4
Evaluation Domain processa
graph.py + nodes.py + state.py — LangGraph interno com InterviewState. Chama Gemini LLM para gerar perguntas e avaliar respostas.
src/domains/evaluation/graph.py → nodes.py → state.py
LangGraph interno InterviewState Gemini LLM FairnessGuard BiasAudit Confidence
↓
5
RubricEvaluation (4 dimensoes)
relevance_score + depth_score + clarity_score + examples_score → overall_score (0-1). Sem framework pedagogico (sem Bloom, sem Dreyfus).
src/domains/evaluation/models.py → RubricEvaluation
4 scores FlowDecision Bloom taxonomy Dreyfus model Big Five
↓
6
Resposta retorna SEM verificacoes
Score + recomendacao ("APPROVED"/"NOT_RECOMMENDED") volta direto ao recrutador. Sem fact-checking, sem bias audit snapshot, sem confidence calibration, sem PII masking dos dados do candidato que foram enviados ao LLM.
src/domains/evaluation/ → response direto
FactChecker BiasAuditSnapshot ConfidenceNode PII demask AuditTrail append-only HITL Gate
RISCOS NESTE FLUXO (v5):
1. Nome, CPF, endereco do candidato foram enviados ao Gemini SEM mascaramento PII
2. Nenhum bias audit — se o LLM discriminou por genero/idade/etnia, nao ha registro
3. Score nao calibrado — sem ConfidenceNode, 0.7 pode significar qualquer coisa
4. Sem auditoria imutavel — registro pode ser alterado/deletado depois
5. Sem fairness check — avaliacao nao passa por FairnessGuard (evaluation e 1 dos 6 dominios sem)
6. Sem fact-checking — claims do candidato nao sao verificados

EnhancedAgentMixin._setup_enhanced() — 16 capacidades injetadas

libs/agents-core/enhanced_agent_mixin.py

↓
4
FairnessGuard filtra ANTES do LLM
3 camadas: (1) Regex — bloqueia "rejeitar por idade/genero"; (2) Implicito — detecta vieses indiretos ("bairro nobre"); (3) LLM — analise semantica de fairness. Se bloqueado → AuditTrail registra + resposta negada.
app/shared/compliance/fairness_guard.py
Layer 1: regex patterns Layer 2: implicit bias Layer 3: LLM analysis
↓
5
PII Masking remove dados sensiveis ANTES do LLM
4 camadas: CPF → [CPF_MASKED], nome → [NAME_1], endereco → [ADDR_MASKED], telefone → [PHONE_MASKED]. O LLM NUNCA ve dados pessoais reais do candidato.
app/shared/compliance/pii_masking.py
CPF masking Name masking Address masking Phone masking
↓
6
WSI Interview Graph processa (1.141 linhas)
Grafo deterministico com 8 stages: INIT → LOAD_CONTEXT → GENERATE_QUESTION → AWAIT_RESPONSE → VALIDATE → SCORE → ADVANCE → COMPLETE. Cada questao com Bloom (1-6) + Dreyfus (1-5) + Big Five (OCEAN). Estado salvo no PostgresSaver (sobrevive a crashes).
app/domains/cv_screening/agents/wsi_interview_graph.py (1141L)
WSIInterviewState Bloom taxonomy (1-6) Dreyfus model (1-5) Big Five (OCEAN) PostgresSaver checkpoint 3 scores: tech + behav + sit
↓
7
Gemini/Claude processa (com dados mascarados)
LLM recebe texto com [CPF_MASKED], [NAME_1], etc. Anti-sycophancy block no system prompt. CircuitBreaker protege contra falha do provider.
Gemini/Claude API (via CircuitBreaker)
AntiSycophancy block CircuitBreaker ModelRouter
↓
8
FactChecker verifica APOS o LLM

Baseado em leitura direta de 820 arquivos v5 (GitHub 19/03/2026) + 1.259 arquivos LIA (lia-agent-system/) + documento de referencia de 5.687 linhas (Paralelo_LIA_vs_V5_Arquitetura_IA.md) com 22 secoes de analise de mercado. 1. Visao Geral, Filosofia & Estrutura (Secoes 1-3)
Por que importa: A escala do codigo reflete maturidade de engenharia. Eightfold AI tem mais de 2.000 arquivos so no core de ML; Ashby tem ~800 com cobertura de testes acima de 70%. A LIA com 1.259 arquivos Python e 313 testes esta no territorio enterprise, enquanto o v5 com 820 arquivos e 96 testes esta no territorio de produto ativo. Phenom separou seu sistema em microservicos por dominio desde a serie B — a LIA segue esse mesmo padrao com 12 dominios, mas o v5 agrupa em 8 dominios sem camada shared real, o que dificulta escalabilidade futura.
LIA — Plataforma SaaS Multi-Tenant
1.259 arquivos Python | 362 endpoints FastAPI | 250+ servicos | 102 modelos SQLAlchemy | 12 dominios especializados | 23 agentes (12 ReAct + 4 StateGraph + 6 subagentes Z1 + PolicyAgent) | 66 tools mapeadas em scope_config | Multi-tenancy nativa com budget + guardrails por empresa | CascadedRouter hierarquico 6 tiers (fast-path → regex → keyword → semantic → LLM → context) | EnhancedAgentMixin garante heranca de compliance para qualquer agente novo.
v5 — Sistema Multi-Agente ATS
820 arquivos Python | REST + WebSocket FastAPI | 51 servicos transversais | PostgreSQL + pgvector + Redis | 8 dominios + autonomous fallback | ~30 agentes/camadas (6 pipeline + 9 sourcing subagentes + domain agents) | 73+ tools no autonomous | Hub 3 camadas (HubOrchestrator → DomainOrchestrator → DomainWorkflow) | CostLadder + ModelRouter (fast/default/heavy Gemini) | Dominios autocontidos sem camada shared de infraestrutura.
Mercado: A tendencia e manter complexidade proporcional ao problema (Anthropic). Paradox (Olivia) usa pipeline simples deliberadamente. Eightfold usa orquestracao hub→domain similar ao v5. A LIA com 6 tiers e mais complexa que ambos, mas a multi-tenancy justifica a complexidade — Workday Recruit tem complexidade similar exatamente porque multi-tenancy e compliance por jurisdicao exigem.
2. Orquestracao & Roteamento (Secao 4)
Por que importa: O roteamento e o cerebro do sistema — um erro de roteamento manda a pergunta para o agente errado, produzindo resposta completamente incorreta. A LIA resolve isso com 6 tiers escalonados (Tier 0 resolve pronomes como "ela" e "aquela vaga" antes do LLM, algo que rarissimas plataformas fazem). O v5 usa CostLadder com fallback para autonomous ReAct — elegante porque se nenhum dominio tem certeza, o autonomous universal resolve. A pratica recomendada pela Anthropic e "least capable model that meets requirements" — exatamente o que o cascade Haiku→Opus da LIA implementa.
✔ LIA
  • CascadedRouter 6 tiers com otimizacao de custo (Haiku→Sonnet→Opus)
  • Tier 0 memory_resolver para pronomes — sofisticacao rara
  • Cache semantico pgvector cosine ≥ 0.92 (configuravel Z5-03)
  • 13 dominios mapeados com PromptRegistry versionado
✔ v5
  • HubPlanner fast-path + CostLadder + LLM = routing otimizado
  • Fallback para autonomous ReAct universal — maxima flexibilidade
  • ModelRouter (fast/default/heavy) otimiza custo single-provider
  • reference_resolver.py existe mas NAO integrado ao Hub
✘ LIA
6 tiers aumentam latencia acumulada e complexidade de debugging — um bug de roteamento pode ter 6 causas diferentes.
✘ v5
Zero otimizacao de custo entre providers (single Gemini). Sem Tier 0 de resolucao de referencia — "ela" e "aquela vaga" vao para o LLM sem pre-processamento, desperdicando tokens e reduzindo precisao.

Por que importa: Memoria e o que diferencia um agente de um chatbot. Sem memoria cross-session, o recrutador precisa repetir tudo no dia seguinte. O v5 tambem tem categorizacao automatica (hiring_preference, workflow_pattern, entity_insight, search_pattern) sem precisar de LLM adicional.

✔ v5

✘ LIA

7. Fairness & Compliance de IA (Secao 11) — PONTO FORTE MAXIMO LIA
Por que importa: NYC Local Law 144 (vigente desde 2023) exige auditoria de disparate impact para sistemas de IA em recrutamento. HireVue foi multado exatamente por nao ter isso. A LIA ja implementa four_fifths_rule() — a regra padrao EEOC/OFCCP que mede se grupos protegidos estao sendo selecionados a uma taxa de pelo menos 80% da taxa do grupo com maior selecao. Fairness como camada 1 (pre-decision) e raro no mercado — a maioria faz fairness post-hoc (depois que o dano ja ocorreu). FRIA documentado e o que a EU AI Act exige para sistemas de IA de alto risco. Isso coloca a LIA a frente de HireVue em compliance.
✔ LIA — FairnessGuard 3 Camadas OBRIGATORIO
  • Camada 1: Pre-Decision Guard → fairness_guard.py → block_if_biased()
  • Camada 2: In-Process Monitor → audit_callback.py → log_decision()
  • Camada 3: Post-Decision Audit → admin_bias_audit.py → four_fifths_rule()
  • LGPD compliance completo: consent_checker, dsr_export, lgpd_cleanup, granular_consent (4 servicos)
  • FRIA documentado (FRIA_WSI.md) — exigencia da EU AI Act
  • Policy domain (Z5-02) para compliance configuravel por tenant
✔ v5 — Fairness em 2 Dominios
  • jobs/fairness.py — verifica vies em job descriptions ("boa aparencia", "jovem")
  • sourcing/fairness.py — verifica vies em filtros de busca de candidatos
  • FactChecker com 3 verificacoes numericas (count, avg, score)
  • InjectionDetector com 20+ padroes PT/EN — testado em test_security.py
  • Anti-hallucination testado: test_hallucination_prevention.py
✘ LIA
Sem red teaming adversarial. Sem relatorio de fairness exportavel para auditores externos (NYC LL144 requer isso). Injection protection implicita, nao tao explicita quanto security.py do v5.
✘ v5
Fairness opt-in (nao obrigatorio) — so em 2 de 8 dominios. Em producao no Brasil/EUA/Europa, risco legal serio. Sem regra 4/5 (EEOC). Sem dashboard de auditoria. Sem LGPD compliance (consentimento, DSR, cleanup).

15. Agent Studio — Comparativo LIA vs v5 & Critica de Escalabilidade
Por que importa: Agent Studio e a capacidade de clientes criarem e configurarem seus proprios agentes sem intervencao do time de engenharia. Eightfold (Project Andromeda) e Phenom (X+ Studio) ja oferecem isso. A arquitetura atual de cada sistema determina a viabilidade e o esforco para implementar Agent Studio. A diferenca fundamental e: a LIA tem infraestrutura compartilhada (EnhancedAgentMixin, PromptRegistry, FairnessGuard como heranca) que torna cada novo agente automaticamente compliant — 0 linhas de implementacao por agente. O v5 tem dominios autocontidos sem camada shared real, o que significa que criar um novo agente exige reimplementar fairness, memory, prompts manualmente — replicacao de codigo desnecessaria que nao escala.
Critica Arquitetural — Escalabilidade do v5 para Agent Studio: Cada dominio do v5 implementa seus proprios prompts.py, fairness.py, memory.py, actions/ de forma isolada. Nao existe src/shared/ com base classes que garantam comportamento padrao. Para um cliente que vai criar varios agentes via Agent Studio, isso resulta em: (1) duplicacao de codigo de fairness, memory e prompts em cada agente novo, (2) impossibilidade de adicionar features transversais (ex: anti-sycophancy) sem modificar N arquivos em N dominios manualmente, (3) risco de agentes novos nao terem compliance porque nao ha heranca automatica. Na LIA, EnhancedAgentMixin resolve isso — qualquer novo agente herda FairnessGuard, tracing, compliance com 0 linhas. O v5 precisa ANTES criar src/shared/ com base classes para viabilizar Agent Studio — caso contrario, cada agente novo criado pelo cliente sera uma copia sem compliance garantido.
